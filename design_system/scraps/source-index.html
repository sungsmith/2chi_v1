이취(2chi)는 내 이력 + 기업분석 + 채용공고를 통합 분석하여 회사별로 다른 자소서를 만들고, 부족한 역량까지 짚어주는 IT 직군용 취업·이직 워크스페이스입니다. 이 문서는 v1 출시를 위한 디자인 시스템으로, 토큰 · 컴포넌트 · 페이지 템플릿 · AI 검토 패턴까지 실제 개발 적용 가능한 단위로 정리되어 있습니다.

00 · Overview

제품 원칙 & 디자인 원칙

이취는 "AI가 대신 써주는 서비스"가 아니라 "사용자가 자신의 경험을 정리하고 검토하는 워크스페이스"입니다. 모든 디자인 결정은 아래 5가지 원칙을 충족해야 합니다.

원칙 01

하나의 흐름으로 연결

이력 → 공고 → 기업분석 → 자소서 → 지원이 단절 없이 이어져 보이게. 같은 데이터를 두 번 입력하지 않는다.

원칙 02

검토 가능한 AI

AI 결과는 항상 "검토 필요" 상태로 시작한다. 출처가 확인되지 않는 표현은 시각적으로 강조하고, 사용자의 확정으로만 최종이 된다.

원칙 03

불안을 낮추는 톤

취업 준비자의 압박감을 키우지 않는다. 부족 역량은 "비난"이 아니라 "보완 액션"으로 안내한다.

원칙 04

감성은 포인트, 정보는 또렷이

테이프 · 메모 · 손글씨는 빈 화면 · 격려 · 환영 영역에만. 자소서 · 이력서 · 데이터 테이블은 가독성 최우선.

원칙 05

신뢰 위에 올린 친근함

민감한 데이터(이력 · 자소서 · 결과)를 다루기에 SaaS급 신뢰감이 기본. 그 위에 친근한 톤을 얇게 얹는다.

v1 SCOPE

포함: 회원/온보딩, 대시보드, 내 정보, 지원 현황, 자소서, 채용공고, 기업분석, 알림, 관리자.
제외: 면접/모의면접, 포트폴리오 빌더, 커뮤니티, 결제, 본인인증, 알림톡/SMS, 네이티브 앱, PWA.

02 · Product Tone & UX Writing

사용자를 압박하지 않는 문장

"AI가 알려준다"가 아니라 "함께 정리해본다". 명령형 대신 청유형 · 권유형으로, 단언 대신 검토 가능한 표현으로 작성합니다.

톤 원칙

✓ 이런 톤으로

  • 해요체 · 청유형 ("작성해볼까요?")
  • 주체는 항상 "내(사용자) "
  • AI 결과는 "초안 · 추천 · 확인 필요"로 표기
  • 부족 역량은 "보완할 경험" / "다음 준비"
  • 구체적 행동을 다음 문장에 함께 제시

✗ 피해야 할 톤

  • 합쇼체 / 명령형 ("작성하십시오")
  • "부족합니다 · 미흡합니다" 같은 평가어
  • "AI가 완성했습니다" 같은 단언
  • "제출하지 않으면 떨어집니다" 같은 위협
  • 이모지 남용 / 감탄사 과다 사용

05B · Icon System

두 개의 아이콘 시스템 — 기능 vs 감성

이취는 두 가지 아이콘 시스템을 명확히 분리합니다. Lucide Icons(기능)와 직접 제작 SVG(장식)는 같은 화면에 함께 등장하지만, 역할이 다르고 사용 위치도 다릅니다.

✓ Functional · 정보 전달

LUCIDE

홈 · 내 정보 · 자소서 · 알림 등 기능을 지칭합니다. 선형 outline, 2px stroke, round cap. 의미는 항상 텍스트 라벨이 보조합니다.

예: Home · UserRound · FileText · CalendarDays · Bell · Search · Sparkles

✨ Decorative · 분위기 형성

CUSTOM SVG

꽃 · 별 · 하트 · 메모지 · 테이프 등 감성을 더하는 장식. 손그림 느낌의 변형된 stroke, 일관되지 않은 곡선. aria-hidden="true"로 처리합니다.

예: 꽃 · 별 · 하트 · 잎 · 새싹 · 강조 밑줄

두 시스템은 같은 카드 안에 함께 등장할 수 있지만, 역할이 섞이면 안 됩니다. 예: "AI 추천" 카드 = Sparkles 아이콘(기능) + 라벤더 구름 마스코트(장식)는 OK. 자소서 본문에 꽃 도들이 들어가는 것은 ✗.

// components/Icon.tsx — 공통 기능 아이콘 래퍼 import type { LucideIcon } from 'lucide-react'; import { cn } from '@/lib/utils'; interface IconProps { icon: LucideIcon; size?: number; // 기본 20 strokeWidth?: number; // 기본 2, soft = 1.75 className?: string; label?: string; // aria-label · 없으면 aria-hidden=true } export function Icon({ icon: IconComponent, size = 20, strokeWidth = 2, className, label, }: IconProps) { return ( <IconComponent size={size} strokeWidth={strokeWidth} className={cn('text-text-primary', className)} aria-hidden={label ? undefined : true} aria-label={label} /> ); } // 사용 예 ─────────────────────────────── <Icon icon={Home} size={18} /> // 사이드바 <Icon icon={Sparkles} className="text-lavender-500" /> // AI <Icon icon={Trash2} label="자소서 삭제" /> // 아이콘 버튼 <Icon icon={FileText} size={48} strokeWidth={1.75} /> // Empty State

장식 SVG는 별도 컴포넌트로 분리. 예: <FlowerDoodle/>, <TapeStripe color="mint"/>. 두 시스템이 같은 import 경로를 공유하지 않도록 폴더를 나눠 관리: components/icons/ vs components/doodles/.

G · 혼용 규칙 · Do / Don't

✓ Do

  • 같은 카드에서 Lucide(기능) + 장식 SVG(분위기)를 함께 사용 — 단, 시각적으로 떨어진 위치에
  • 아이콘 옆에 항상 텍스트 라벨 또는 aria-label
  • Empty / Onboarding / CTA에서는 stroke 1.75 + 큰 사이즈로 부드럽게
  • 의미 색은 시맨틱 토큰으로 (성공=mint, 경고=yellow, 오류=red, AI=lavender)
  • 장식 SVG는 카드의 모서리 · 빈 공간에만
  • 아이콘 색은 옆 텍스트보다 약하게 (대비 강하지 않도록)

✗ Don't

  • 장식 SVG로 기능을 지칭 (예: 꽃 도들 = "추가" 버튼) — 금지
  • 아이콘만으로 의미 전달 (필수 텍스트 라벨 누락)
  • 자소서 본문 · 테이블 셀 · 관리자 화면에 장식 SVG 사용
  • 경고 / 오류 메시지 옆에 캐릭터 · 메모지 · 테이프 배치
  • Lucide 아이콘을 색상만 바꿔 장식 용도로 활용
  • 한 화면에 장식 요소 3개 이상
  • 소셜 로그인 로고 색상 변경 — 공식 가이드 무시

10 · Accessibility

WCAG 2.1 AA 기준 적용

민감한 데이터를 다루는 서비스이므로 접근성은 선택이 아니라 필수입니다. 모든 인터랙티브 요소는 키보드만으로 조작 가능해야 합니다.

텍스트 대비

본문 텍스트≥ 4.5:1 (AA)
큰 텍스트 (18pt+)≥ 3:1
UI · 아이콘≥ 3:1
Disabled (예외)대비 제한 면제

text.muted (#8A8980) on bg는 4.5:1 미달이므로 본문에 사용 금지. 캡션/메타에만.

키보드 이동

Tab다음 인터랙티브 요소
Shift + Tab이전
Esc모달 · 팝오버 닫기
Enter / Space버튼 실행
←→탭/캘린더 이동

Focus Ring

모든 인터랙티브 요소는 :focus-visible 시 3px 외곽 링 표시.

색상 + 외곽 두께 동시 변화. 색상만으로 표시하지 않음.

폼 에러 안내

색상 + 아이콘 + 텍스트 3중 표시. aria-describedby로 에러 메시지 연결.

✗ 올바른 이메일 형식이 아니에요.

아이콘 단독 사용 금지

아이콘 버튼은 반드시 aria-label 또는 툴팁.

aria-label="알림 보기"

색상만으로 상태 구분 금지

전형 단계 색상은 항상 텍스트 라벨과 함께. 도트(●) + 텍스트 조합.

서류 작성중 1차 면접 합격 불합격

스크린리더용 라벨

아래 영역은 SR 전용 텍스트 권장.

  • 매칭률 차트 → aria-label="JD 매칭률 82%"
  • 거짓방지 하이라이트 → aria-label="이력에서 확인되지 않은 표현"
  • 마스코트 / 도들 → aria-hidden="true"

알림 · 토스트 접근성

토스트는 role="status" (정보) 또는 role="alert" (에러). 시간이 지나도 자동으로 사라지지 않는 옵션 제공 (≥ 5초).

11 · Usage Guidelines

레이아웃 · 컬러 · 텍스트 사용 원칙

아래는 디자이너 · 개발자가 화면을 만들 때 반복 결정해야 할 항목의 기준입니다.

💡 정보 위계

  • 한 화면에 H1 / Page Title은 1개
  • 주요 액션 (Primary Button)도 1개
  • Section Title은 본문보다 위에, Card Title은 카드 좌상단
  • 데이터(숫자)는 텍스트보다 크게, 색은 강조 마지막에

📐 카드 단위 그룹화

  • 관련 정보는 카드 안에 묶기 (테두리 1px + radius lg)
  • 카드 내부 패딩 기본 20px, 모바일 16px
  • 카드끼리 사이 16px gap
  • 카드 안에 또 다른 카드 깊이 2단계까지만

🎨 색상 의미

Primary (Periwinkle)핵심 액션, 현재 위치

✏️ 마이크로카피

  • 해요체 · 청유형 · 1인칭("내")
  • 버튼: 동사로 시작 (✓ "저장하기" · ✗ "저장")
  • 금지된 단어: 부족 / 미흡 / 실패 / 잘못된
  • 권장 단어: 보완할 / 다음 준비 / 검토 / 확인

⚡ 인터랙션

  • 호버 · 액티브 0.04~0.2s · ease-out
  • 모달 진입 0.2s scale 0.96 → 1 + fade
  • 토스트 4s 후 자동 dismiss, 에러는 수동 dismiss
  • 스켈레톤 1.5s shimmer · prefers-reduced-motion 존중

📊 데이터 / 숫자 표기

  • 모든 숫자는 JetBrains Mono
  • 매칭률 / 진행률은 % 기호 작게(0.6em)
  • D-day는 핑크 100/400 단일 토큰
  • 날짜 표기: 2026.05.20 (점 구분, 2자리)
12 · Do / Don't

해야 할 것 · 하지 말 것

아래 항목은 디자인 리뷰 체크리스트로도 사용됩니다. 새 화면을 그릴 때 마지막으로 한 번 더 확인해주세요.

✓ Do

  • 정보 위계를 명확히 — H1, Primary Btn 각각 1개
  • 카드 단위로 정보 그룹화 (radius lg + soft shadow)
  • 감성 요소(테이프 · 메모 · 손글씨)는 환영 · CTA · 빈 화면에만
  • 상태 색상은 의미에 맞게 (Mint=성공 · Yellow=검토 · Red=오류)
  • AI 결과는 항상 "검토 가능한 형태"로 (하이라이트 + 버튼)
  • 부족 역량은 "보완 액션"으로 부드럽게 안내
  • 자소서 작성 화면은 글쓰기 집중 우선 (장식 최소)
  • 키보드만으로 모든 핵심 흐름 완수 가능
  • 아이콘은 텍스트 라벨 또는 aria-label 동반
  • 다크 텍스트(#25241F) + 크림 배경(#F8F8F4) 조합 유지

✗ Don't

  • 여백 없이 빽빽한 배치 (카드 사이 16px 미만 금지)
  • 장식 요소 과다 사용 (한 카드에 테이프 3개 이상 금지)
  • 낮은 대비 텍스트 (text.muted on bg, 본문 사용 금지)
  • AI 결과를 "정답"처럼 표현 ("자동 완성됨" · "검토 불필요" 금지)
  • 경고 메시지를 공격적으로 ("실패!" · "잘못됨!" 금지)
  • 자소서 · 이력서 본문에 캐릭터 / 도들 사용
  • 관리자 화면에 메모지 · 손글씨 · 마스코트 사용
  • v1 범위 밖 기능(면접/커뮤니티/결제)을 메인 동선에 노출
  • 색상만으로 상태 구분 (반드시 아이콘 + 텍스트 동반)
  • 비비드 그라데이션 · 비비드 배경 위 로고 사용
