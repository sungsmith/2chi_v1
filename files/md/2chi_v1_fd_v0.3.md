# 1. 프로젝트 개요

## 1.1 프로젝트명

2chi (한글명: 이취 / 이직 + 취직)

## 1.2 목적 및 배경

취업·이직 준비자가 자소서 생성, 입사 지원 관리, 부족한 경험에 대한 컨설팅까지 한 플랫폼에서 처리할 수 있는 올인원 워크스페이스를 제공한다. 본 프로젝트는 기획자가 본인의 이직 준비 과정에서 느낀 페인포인트(여러 도구 간 컨텍스트 복붙 반복, 회사별 자소서 재작성 부담, 지원 일정·결과 추적 부재 등)를 해소하는 것을 목적으로 한다.

## 1.3 한 줄 정의

내 이력 + 기업분석 + 채용공고를 통합 분석하여 회사별로 다른 자소서를 생성하고, 부족한 역량까지 짚어주는 IT 직군 신입·중고신입용 이직/취업 워크스페이스.

## 1.4 핵심 컨셉

- 사용자의 이력·경력과 이직/취업 희망 기업·채용공고를 매칭하여 맞춤 자소서·경력기술서·포트폴리오 정리물을 생성한다.
- 같은 사용자라도 회사 A와 B에 지원할 때 자소서가 다르게 생성된다. (기업분석 + JD 차이를 반영)
- 이력에 없는 경험·역량을 자소서에서 임의 생성하지 않는다. (Hallucination guard)
- 부족한 경험은 컨설팅으로 안내한다. (직무별 표준 역량 매트릭스 기반 갭 분석)

## 1.5 차별점

- 컨텍스트 자동 통합: 사용자가 이력·공고·기업분석을 매번 복붙하지 않아도 한 번의 클릭으로 통합된 컨텍스트로 자소서가 생성된다.
- 회사별 자동 차별화: 같은 이력이라도 기업분석 결과와 JD 차이에 따라 자소서 톤·강조점이 자동 조정된다.
- 역량 컨설팅: 단순 자소서 생성에 그치지 않고, 희망 포지션 표준 역량과 보유 역량의 갭을 분석하여 보완 액션을 제안한다.
- 올인원 지원 관리: 자소서 생성에서 캘린더/대시보드/히스토리까지 한 플랫폼에서 처리된다.
- 거짓 방지 가드: 이력에 없는 내용으로 자소서가 작성되면 작성 후 검토 화면에서 강조 표시한다.

- 포트폴리오 풀 빌더 (v1은 외부 링크 모음 수준만 지원)

- 전문 첨삭 마켓 / 헬퍼 매칭
- 본인 인증 (휴대폰/아이핀)
- 카카오 알림톡, SMS 발송
- 결제 / 구독 / 광고 (수익화 기능)
- 네이티브 모바일 앱 (iOS/Android) — v2 직전 별도 개발 예정
- PWA (모바일은 URL 접속으로 해결)

## 3.3 v1 운영 형태

- 클로즈드 베타: 일반 공개하지 않고 기획자 본인 + 지인 N명 대상 회원가입 개방.
- 배포는 v2부터, v1은 개발 완료 + 본인 사용 + 지인 피드백 수집을 목표로 한다.
- 팀 구성: 기획/백엔드 1명(김소미), 프론트/디자인 1명, Claude Code 적극 활용.

## 4.2 환경 분리

- local: 개발자 PC, Docker Compose
- staging(v1): 본인 PC + Cloudflare Tunnel로 지인 테스트용 공개 URL 발급. 본인 PC 가동 시간 동안만 운영.
- production(v2): 클라우드 배포 시점에 결정. v1 단계는 staging만 운영.

## 4.3 모바일 대응

v1은 PC 웹 우선 + 반응형 디자인으로 모바일 브라우저 접속 시 기본 사용 가능. PWA 미적용. 네이티브 앱(React Native 또는 Flutter)은 v2 진입 직전 별도 프로젝트로 개발.

## 4.4 v2 이전 용이성 원칙

- S3 호환 인터페이스 추상화: v1은 MinIO, v2는 클라우드 스토리지 — 코드 변경 없이 endpoint만 교체
- 환경변수 기반 설정: API 키·DB 접속·외부 URL 등 모두 .env로 관리, 코드에 하드코딩 금지
- 도커 컨테이너 단위 운영: v1부터 모든 컴포넌트(웹/API/DB/MinIO)를 Docker Compose로 구성
- 외부 의존성 어댑터 패턴: 메일·LLM·인증 등은 인터페이스로 추상화하여 구현체 교체 용이
- DB 스키마 마이그레이션 도구 사용: Flyway 또는 Liquibase로 v2 이전 시 스키마 일관성 보장
- Kubernetes 미사용: v1·v2 모두 단일 컨테이너 운영. v3 이후 트래픽 증가 시 검토.
- CI/CD는 v1부터 도입: GitHub Actions로 푸시 시 자동 테스트 + Docker 이미지 빌드. 실제 배포 자동화는 v2 클라우드 결정 후 단계별 도입.

## 5.1 회원 / 계정

### 5.1.1 회원가입

- 이메일 + 비밀번호 회원가입
- 소셜 회원가입: 카카오, 네이버, 구글 (OAuth2)
- 필수 동의: 서비스 이용 약관, 개인정보 수집·이용 동의, 만 14세 이상 확인
- 선택 동의: 마케팅 정보 수신
- 이메일 인증: 가입 후 인증 메일 발송, 24시간 이내 클릭

#### 입력 항목

| 항목 | 필수 | 검증 |
| --- | --- | --- |
| 이메일 | Y | RFC 5322 형식, 중복 불가 |
| 비밀번호 | Y(이메일 가입) | 8자 이상, 영문/숫자/특수문자 중 2종 이상 조합 |
| 닉네임 | Y | 2~20자, 한/영/숫자, 중복 불가 |
| 만 14세 이상 확인 | Y | 체크박스 |

#### 예외 처리

- 이메일 중복 → '이미 가입된 이메일입니다. 로그인 화면으로 이동하시겠어요?' 안내
- 비밀번호 정책 미달 → 입력 즉시 실시간 검증 메시지
- 미성년자 → '만 14세 미만은 가입할 수 없습니다' 메시지로 차단

### 5.1.2 로그인 / 로그아웃

- 이메일 + 비밀번호 / 소셜 로그인
- 로그인 유지: Refresh Token 14일, Access Token 30분
- 로그인 5회 실패 시 10분간 잠금
- 로그아웃: 클라이언트 토큰 폐기 + 서버 Refresh Token 블랙리스트 처리

### 5.1.3 비밀번호 재설정

- 이메일 입력 → 재설정 링크 전송 (1시간 유효)
- 재설정 시 기존 비밀번호와 다른지 검증

### 5.1.4 마이페이지 (계정 설정)

- 닉네임 변경
- 비밀번호 변경 (이메일 가입자)
- 연결된 소셜 계정 관리 (연결/해제)
- 알림 설정 (이메일/웹푸시 ON/OFF, 카테고리별)
- 회원 탈퇴: 30일 유예 후 영구 삭제 (이력서·자소서 포함)

## 5.2 온보딩

회원가입 직후 1회 진행되는 4단계 가이드. 스킵 가능하나 미입력 시 첫 화면에서 재안내.

### 5.2.1 단계별 화면

## 5.6 이직 / 취업

### 5.6.0 직군별 권장 작성 구조 (v0.3 신규)

자소서·경력기술서 작성 시 사용자의 희망 직군에 맞춰 권장 구조가 자동 적용된다. 마스터 자소서·AI 초안·경력기술서 입력 폼 모두에 동일 구조가 반영된다. 인프라/DevOps 직군은 사용자 경력 키워드에 따라 두 트랙 중 한쪽으로 자동 판정한다.

| 직군 | 권장 구조 | 핵심 포인트 및 예시 |
| --- | --- | --- |
| 백엔드 개발자 | PRAR 문제(Problem) → 원인(Root cause) → 접근(Approach) → 결과(Result) | • 트래픽·성능·장애 케이스에 강함 • 정량 수치(TPS·지연시간·처리량·비용) 필수 • 예: TPS 500 한계 → N+1 쿼리 발견 → 인덱스+쿼리 최적화 → TPS 2,000 |
| 프론트엔드 개발자 | UX-Driven 사용자/UX 문제 → 기술 선택·접근 → 구현 결정 → 사용자 임팩트 | • 정량 결과(이탈률↓·FCP/LCP↓·전환율↑) + 인터랙션 개선 • 디자이너와의 협업 케이스 매력적 • 예: 모바일 이탈률 18% → LCP 4.2s 측정·Next.js Image 도입 → LCP 1.6s, 이탈률 9% |
| 인프라 / DevOps (2개 트랙) | Ops-Result 운영 문제·병목 → 측정·원인 분석 → 자동화·도구 도입 → 안정성·효율 결과 · 클라우드 트랙: AWS / k8s / Terraform / CI·CD · 시스템 운영 트랙: VMware / NAS / 백신 / 백업 / 정보보안 / 네트워크 / 랜섬웨어 대응 | • MTTR·배포 빈도·가용성(SLO)·RPO/RTO·비용 절감·재발률 • 클라우드 예: 배포 4시간 수동 → GitHub Actions + ArgoCD 도입 → 15분, 주 1회 → 일 5회 • 시스템 운영 예: 영업팀 PC 5대 랜섬웨어 감염 → EDR + 일 증분 백업 도입 → 데이터 100% 복구, 다운타임 4시간, 재발 0건 • 시스템 운영 예: 물리서버 12대 활용률 25% → VMware ESXi 통합 → 3대로 통합, 활용률 70%, 전력·라이선스 40% 절감 |
| UI/UX 디자이너 | Design Thinking 4단 사용자 문제 정의 → 인사이트(리서치) → 디자인 솔루션 → 검증(A/B·정량 결과) | • Double Diamond 또는 5단 디자인 씽킹 변형 • 검증 단계 정량 결과(전환율·만족도·태스크 완료율) 필수 • 예: 결제 페이지 이탈률 32% → 사용자 인터뷰 8건·휴리스틱 평가 → 3단계 단순화 → 이탈률 12% |

### 5.6.1 자소서

#### (0) 마스터 자소서 (v0.2 신규)

- 페르소나 B의 핵심 워크플로우 지원: '마스터 자소서 한 벌 → 회사별 변형 제출'
- 항목 유형별로 사용자의 마스터 답변을 한 번만 잘 작성해두고, 회사 지원 시 AI가 기업분석 + JD를 적용해 변형본을 자동 생성한다.
- 항목 유형(v1): 지원동기, 입사 후 포부, 협업 경험, 갈등 해결, 성취 경험, 문제 해결, 본인의 강점, 본인의 약점, 기타
- 개발자 직군 마스터 답변은 PRAR(문제-원인-접근-결과) 구조 권장 (디자이너는 별도 구조)
- 메뉴 위치: 이직/취업 > 자소서 > 마스터 탭
- 마스터는 회사·공고와 무관하게 사용자 단위로 저장. 회사별 변형본은 별도 레코드 (5.6.1 (5) 참조).

#### (1) 작성 화면 구성

- 자소서 항목 입력란 (제목 + 글자수 제한 입력) — 필수
- 채용공고 연결 (드롭다운 또는 검색) — 필수
- 기업분석 연결 (자동 채택 또는 수동 선택) — 선택
- 마스터 답변 자동 연결 (항목 유형 일치 시 자동 채택, 수동 변경 가능) — 선택
- 사용자 요청사항 (자유 텍스트) — 선택
- AI 초안 생성 버튼
- 좌·우 분할 영역 — 좌측 'AI 초안(읽기 전용)' / 우측 '수정본(편집 가능)'. 사용자가 비교하며 수정.
- 사이드 패널: 글자수 카운터, 맞춤법 검사 결과, 거짓방지 검토 표시

#### (2) AI 초안 생성

- 입력: 내 정보(이력+경력+포트폴리오 메타데이터) + 마스터 답변(해당 항목 유형 있을 시) + 채용공고 + 기업분석 + 항목·글자수 + 사용자 요청
- 프롬프트 분기 (이중 축)
  - 축 1 — 사용자 희망 직군: 5.6.0 직군별 권장 구조 적용 (PRAR / UX-Driven / Ops-Result / Design Thinking)
  - 축 2 — 자소서 항목 유형: 지원동기·입사 후 포부·협업 경험·갈등 해결·성취 경험·문제 해결 등 자동 인식 후 유형별 템플릿 적용
- 인프라/DevOps는 사용자 경력 키워드로 클라우드 트랙 / 시스템 운영 트랙 자동 판정
- 거짓방지(Hallucination guard) 적용: 사용자 이력에 명시된 회사·프로젝트·기술·성과 외에는 새로 만들지 않음
- 생성 후 자동 검증: 글자수 정확, 공고 핵심 키워드 N개 이상 반영(기본 N = max(3, K×30%), K=공고 핵심 키워드 총 수), 이력에 없는 단어 사용 시 경고

#### (3) 글자수 / 맞춤법

- 실시간 글자수 카운터 (공백 포함/제외 토글, 항목별 제한 초과 시 빨간색)
- 맞춤법 검사: LLM 호출 기반(별도 트리거 버튼). v1은 LLM 검사로 운영, 서비스화 단계에서 부산대 검사기 라이선스 검토.

#### (4) 거짓방지 검토 표시

- 작성 완료된 자소서에서 사용자 이력에 명시되지 않은 고유명사/기술명/성과를 강조 표시
- 표시된 부분에 호버 시 '이 표현은 사용자 이력에서 직접 확인되지 않습니다' 안내
- 사용자가 검토 후 수동 확정

#### (5) 저장 / 다운로드 / 삭제

- 저장 단위: 마스터(사용자 × 항목 유형) / 회사별 변형본(사용자 × 채용공고 × 항목 유형)
- v1은 둘 다 버전 관리 없이 단일 최신본만 저장 (단순화). 버전 관리는 v2.
- AI 초안과 사용자 수정본은 같은 변형본 레코드 안에 별도 컬럼으로 보관 (AI 초안은 참조용으로 잔존)
- 다운로드: PDF, Word (수정본 기준)
- 삭제: 휴지통 30일 보관 후 영구 삭제

### 5.6.2 경력기술서

- 내 정보의 경력 입력 내용을 채용공고·기업분석·사용자 요청 기준으로 재구조화
- 재구조화 결과를 사용자가 수정 가능
- 저장 / 다운로드(PDF, Word)
- v1은 경력기술서도 단일 최신본만 저장

## 5.7 기업

### 5.7.1 채용공고

#### (1) 등록 방식 (v1)

- URL 입력 → 메타데이터 파싱 시도 → 실패 시 본문 직접 붙여넣기 폴백
- 직접 작성: 회사명, 직무, 주요 업무, 자격 요건, 우대 사항, 마감일, 출처 URL
- v1에서는 사람인 / 잡코리아 OpenAPI 검색·집계 화면을 제공하지 않음 (v2로 이동)

#### (2) 저장 데이터

- v1은 메타데이터(회사/제목/직무/마감일/URL) + 자격요건/우대사항 텍스트만 저장. 본문 전체 영구 저장은 저작권 관점에서 최소화.

#### (3) JD 매칭

- 자소서 작성 시 채용공고를 연결하면 사용자 이력의 키워드와 JD 키워드를 자동 매핑하여 매칭률(%) 표시
- 부족한 키워드는 '이 공고는 [Kafka, MSA] 경험을 우대합니다. 이력에 보완을 추천합니다' 안내

### 5.7.2 기업분석

#### (1) 시작

- 기업명 입력 → 동명 기업 존재 시 후보 리스트 → 선택
- DART 미공시 / 정보 부족 시 '정보가 제한적입니다' 안내 + 사용자 직접 입력 옵션

#### (2) 분석 결과 구성

- 회사 개요: 사업 영역, 주요 제품/서비스, 규모(매출/인원)
- 최근 이슈: 최근 6개월 뉴스 키워드 (수주/신사업/위기/리더십 변경 등)
- 인재상: 회사 홈페이지 채용 페이지에서 추출
- 채용 절차: 알려진 절차(서류→코테→면접 등)
- 자소서·면접 활용 포인트: '이 회사는 ○○를 강조하므로 본인 이력의 ○○ 경험을 ○○ 식으로 풀면 효과적' 형태의 액션 제안

#### (3) 데이터 소스

- DART 공시 API (재무/임원/사업 보고서)
- 뉴스: 네이버 뉴스 검색 API 또는 RSS
- 회사 홈페이지 크롤링: 회사 도메인 한정, robots.txt 준수

#### (4) 결과 형식

- 내부적으로는 텍스트(자소서 생성 컨텍스트로 활용)
- 화면 표출은 항목별 카드/섹션으로 정리

# 8. 비기능 요구사항

## 8.1 보안

- 전 구간 HTTPS (TLS 1.2+)
- 비밀번호: bcrypt(cost 12+) 해시
- 민감 데이터(이력서/자소서/연락처): DB 컬럼 단위 AES-256 암호화 (또는 RDS 저장 암호화 + 애플리케이션 레이어 마스킹)
- 토큰: Access JWT 30분, Refresh 14일, Refresh 회전(Rotation) + 블랙리스트
- CSRF: 쿠키 사용 시 SameSite=Strict + CSRF 토큰
- XSS: 출력 시 자동 이스케이프 + CSP 헤더
- SQL Injection: ORM/PreparedStatement만 사용, raw 쿼리 금지
- Rate Limit: 로그인/회원가입/AI 호출 IP 단위 분당 제한
- LLM 호출 시 PII 마스킹 (전화/주민번호 등 정규식 검출 후 토큰 치환) — v2 강화 권장

## 8.2 성능

- 페이지 First Contentful Paint < 2초 (PC 기준)
- 자소서 AI 생성 응답 < 30초 (실패 시 재시도 1회)
- DB 인덱스: User.email(unique), JobPosting(company, deadline), Application(user_id, stage), Notification(user_id, sent_at)
- 자소서 생성 결과는 사용자 단위 캐시 1시간 (동일 입력 재요청 방지)

## 8.3 가용성

- v1은 단일 인스턴스 운영 가능. 일일 1회 DB 자동 백업(7일 보관)
- LLM API 장애 시 사용자에게 'AI 응답이 지연됩니다. 잠시 후 다시 시도해주세요' 표출

## 8.4 로깅

- 구조화 JSON 로그(level, ts, trace_id, user_id, action, latency)
- 파일/DB 저장 대신 표준출력 → 클라우드 로그 서비스 수집
- 로그 보관: 30일 (v1)
- 개인정보(이름/이메일/이력 본문)는 로그에 기록 금지

## 8.5 에러 처리

- 도메인 예외(BusinessException) vs 인프라 예외 분리
- 글로벌 예외 핸들러(@ControllerAdvice)에서 표준 에러 응답으로 변환: { code, message, traceId }
- 4xx는 사용자 안내, 5xx는 사용자에게 상세 미노출 + 운영자 알림
- 프론트 에러 페이지: 404, 500 전용 화면 + 홈으로 이동 CTA

## 8.6 유효성 검증

- 프론트: 입력 즉시 실시간 검증 (UX 향상)
- 백엔드: Bean Validation(JSR-380) + 비즈니스 규칙 검증
- 이중 검증 원칙 (프론트만 신뢰하지 않음)

## 8.7 권한

- 역할: USER(일반 사용자), ADMIN(운영자)
- v1은 RBAC 단순 적용, ABAC는 v2 검토
- 자원 단위 권한: 사용자는 본인 데이터만 접근/수정

## 8.8 네이밍 규칙

- Java 패키지: com.twochi.{domain}
- REST URI: 복수형 명사 (예: /api/v1/cover-letters/{id})
- DTO: {Resource}Request, {Resource}Response
- DB 테이블: snake_case 단수형 (예: cover_letter)
- 프론트 컴포넌트: PascalCase, 훅은 useXxx, 파일명 kebab-case

# 9. 법적 / 컴플라이언스

## 9.1 v1 (클로즈드 베타) 단계

- 약관 / 개인정보처리방침 게시 (간소화 버전이라도 반드시 포함)
- 회원가입 시 동의 화면 + 동의 로그 보관
- 이력서·자소서 저장 시 사용자에게 'AI 학습/모델 개선에 사용하지 않음'을 명시
- LLM API 호출 시 데이터 전송에 대한 동의 명시

## 9.2 서비스화(v2 이전) 단계 추가 사항

- 개인정보 보호책임자(CPO) 지정 및 공지
- 정보통신망법 / 개인정보보호법 점검 (수집·이용·보관·파기 절차 문서화)
- 본인 인증 도입 검토
- 사람인/잡코리아 등 외부 데이터 사용에 대한 약관 재확인 또는 파트너십 협의
- 저작권 정책: 자소서 결과물의 권리는 사용자, 플랫폼은 모델 개선 외 비공개 사용권만 보유 — 약관 명시
- 크롤링 대상 도메인 화이트리스트 + robots.txt 준수 확인

## 9.3 채용공고 본문 처리 원칙

- 메타데이터(회사/제목/직무/마감/URL)만 영구 저장
- 본문 전문은 사용자 화면에서만 가공·표시하고 영구 저장은 최소화
- 사용자가 직접 입력한 본문은 사용자 본인 화면에서만 활용

# 부록 A. 용어집

| 용어 | 정의 |
| --- | --- |
| JD (Job Description) | 채용공고 안의 '주요 업무 / 자격 요건 / 우대 사항' 부분. 직무기술서. |
| JD 매칭 | 사용자 이력의 키워드·경험과 JD 요구사항을 매핑하여 매칭률을 산출하는 기능. |
| 거짓방지(Hallucination guard) | AI가 사용자 이력에 없는 사실을 자소서에 만들어내지 않도록 하는 가드레일. |
| 중고신입 | 1~2년 이내 짧은 경력 보유자가 신입 또는 경력으로 지원하는 형태. |
| 스캔 PDF | 종이 문서를 스캔해 이미지로 만든 PDF. 텍스트 레이어가 없어 텍스트 추출 불가. |
| 표준 역량 매트릭스 | 직무별로 요구되는 역량을 정형화한 데이터셋. 컨설팅·갭 분석에 활용. |
| 클로즈드 베타 | 일반 공개 없이 초대된 사용자만 접근 가능한 테스트 단계. |
