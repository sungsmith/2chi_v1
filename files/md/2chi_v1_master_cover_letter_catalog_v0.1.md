# 1. 개요

본 문서는 2chi v1의 마스터 자소서 기능에서 사용되는 9개 항목 유형을 정의하고, 각 유형마다 직군별 작성 전략·샘플·AI 프롬프트 가이드를 제공한다. CoverLetterMaster 엔티티의 item_type 컬럼이 본 카탈로그의 코드를 그대로 사용한다.

## 1.1 9개 항목 유형 (item_type enum)

| 코드 (DB enum) | 한글명 | 글자수 가이드 | 직군 차이 영향 |
| --- | --- | --- | --- |
| MOTIVATION | 지원 동기 | 300~700자 | 큼 |
| FUTURE_PLAN | 입사 후 포부 | 300~700자 | 큼 |
| TEAMWORK | 협업 경험 | 400~800자 | 중간 |
| CONFLICT | 갈등 해결 경험 | 400~700자 | 적음 |
| ACHIEVEMENT | 성취 경험 | 500~1000자 | 큼 |
| PROBLEM_SOLVING | 문제 해결 경험 | 500~1000자 | 큼 |
| STRENGTH | 본인의 강점 | 300~600자 | 중간 |
| WEAKNESS | 본인의 약점 | 300~600자 | 적음 |
| OTHER | 기타 (자유 형식) | 공고별 상이 | 사용자 정의 |

| --- | --- |

| 프론트엔드 | UX-Driven (UX 문제 → 기술 접근 → 구현 → 임팩트) |

| UI/UX 디자이너 | Design Thinking 4단 (사용자 문제 → 인사이트 → 솔루션 → 검증) |

## 1.3 사용 방법

- v1 마스터 자소서 작성 화면에서 항목 유형 선택 시 본 카탈로그의 가이드가 도움말로 표출
- AI 초안 생성 시 본 카탈로그의 [AI 프롬프트 핵심 지시] 섹션이 시스템 프롬프트의 일부로 주입됨
- 작성 후 자동 검증의 일부로 [흔한 실수] 항목이 경고 트리거로 활용됨 (선택적 적용)
- 샘플은 사용자에게 직접 노출하지 않고 AI few-shot 컨텍스트로만 활용 (사용자 모방·복제 방지)

## 1.4 문서 구조 (각 유형 공통)

- 메타데이터 표
- 이 항목이 묻는 것 / 평가 포인트

- 마스터 답변 샘플 (1~3개 직군)
- 흔한 실수 / 안티 패턴

# 2. MOTIVATION — 지원 동기

## 2.1 메타데이터

| 속성 | 값 |
| --- | --- |
| DB enum | MOTIVATION |
| 한글명 | 지원 동기 |
| 권장 글자수 | 300~700자 |
| 직군 영향 | 큼 (회사 도메인·문제와 직군 경험을 직결) |
| 채용 담당자가 보는 것 | 회사 이해도 · 본인 경험 연결 · 진정성 · 회사 핵심 키워드 반영 |

## 2.2 이 항목이 묻는 것

"왜 우리 회사에 지원했는가"를 통해 (1) 회사·도메인에 대한 이해도, (2) 본인 이력과 회사 도전과제의 연결, (3) 단순 '취업을 위해서'가 아닌 진정성 있는 동기를 본다. 채용 담당자는 이 항목에서 회사 조사를 했는지·복붙 자소서인지를 가장 빠르게 판별한다.

## 2.3 직군별 작성 전략

- 백엔드: 회사가 풀고 있는 기술 문제(MSA 전환·대용량·결제 정합성 등) → 본인의 PRAR 경험 1건과 직결
- 프론트엔드: 회사의 사용자 경험 과제 → 본인의 UX 개선 경험 (정량 결과 포함)
- 인프라/DevOps 클라우드: 회사의 인프라 스케일·자동화·보안 과제 → 본인의 Ops-Result 경험
- 인프라/DevOps 시스템 운영: 회사의 안정성·보안·운영 효율 과제 → 본인의 안정성 개선·사고 대응 경험
- UI/UX 디자이너: 회사의 사용자 문제·서비스 임팩트 → 본인의 Design Thinking 검증 경험

## 2.4 마스터 답변 샘플

▶ 백엔드 (PRAR) 샘플  ·  (주)테크컴퍼니 결제·정산 백엔드  ·  487자

(주)테크컴퍼니의 결제·정산 도메인에 합류하고 싶습니다. 회사는 최근 MSA 전환과 글로벌 결제 게이트웨이 통합을 추진 중이고, 제가 직전 회사에서 해결한 정산 시스템 병목 문제와 직접 연결됩니다.

[P] 직전 회사의 주문 정산은 월말 마감마다 TPS 500 한계로 10분 이상 지연되었습니다.
[R] 분석 결과 N+1 쿼리, 단일 DB 직접 접근, 동기 처리 의존이 원인이었습니다.
[A] JPA 패치 전환, 복합 인덱스 도입, Kafka 비동기 처리를 적용해 정산 도메인을 점진적으로 분리했습니다.
[R] TPS 500 → 2,000, 처리 지연 10분 → 30초, 월 운영비 200만원 절감을 달성했습니다.

이 경험은 귀사의 MSA 전환과 결제 정합성 과제에 그대로 적용할 수 있다고 판단합니다. 특히 글로벌 결제 게이트웨이 통합에서 핵심이 될 이벤트 기반 정합성 보장에 기여하고 싶습니다.

▶ 인프라 시스템 운영 (Ops-Result) 샘플  ·  (주)미들마켓 IT인프라팀  ·  478자

(주)미들마켓의 IT 인프라팀에 합류하고 싶습니다. 최근 ISMS-P 인증을 추진하면서 백업·재해복구 체계를 재정비하고 있다는 점을 채용 페이지와 보도자료에서 확인했고, 직전 회사에서 제가 주도한 일과 정확히 같은 과제입니다.

[측정] 백업 로그·복구 테스트 결과를 분석해 RPO 24시간·RTO 8시간으로 정량화했습니다.
[자동화] Veeam 도입으로 일 증분·주 풀백업·매주 자동 복구 검증을 운영하고, 오프사이트(외장 디스크) 정책을 추가했습니다.
[결과] RPO 1시간·RTO 30분으로 개선했고, 이후 1년간 백업 실패 0건·복구 훈련 100% 통과를 유지했습니다.

귀사의 ISMS-P 추진에 백업·복구 표준화로 기여하고 싶습니다.

(주)헬스테크의 사용자 경험 개선 미션에 공감해 지원합니다. 채용 페이지에서 'B2C 헬스 서비스의 30대 사용자 유지율 개선'이 핵심 과제라고 명시하셨는데, 직전 회사 헬스앱에서 제가 직접 해결했던 문제와 일치합니다.

[사용자 문제] 식단 기록 화면에서 30대 사용자의 14일 리텐션이 22%로 낮았습니다.
[인사이트] 사용자 인터뷰 8건에서 '식단 입력이 매번 7단계 이상으로 번거롭다'는 공통 인사이트를 도출했습니다.
[솔루션] 자주 먹는 메뉴 빠른 입력 + 사진 인식 자동 등록을 우선순위로 정의해 3단계로 단순화했습니다.
[검증] A/B 테스트에서 30대 14일 리텐션이 22% → 41%로 개선되었습니다.

같은 접근으로 귀사 서비스의 사용자 유지율을 개선하는 데 기여하고 싶습니다.

## 2.5 흔한 실수

- "성장할 수 있는 회사라서" 같은 막연한 표현
- 회사 홈페이지/공지 복사 ("창립 20년의 글로벌 기업")
- 본인 이력에 없는 경험을 만들어내기 (Hallucination)

- 지원하지 않은 직군의 이야기 (백엔드 지원에 PM 경험 강조)

# 3. FUTURE_PLAN — 입사 후 포부

## 3.1 메타데이터

| 속성 | 값 |
| --- | --- |
| DB enum | FUTURE_PLAN |
| 한글명 | 입사 후 포부 |
| 권장 글자수 | 300~700자 |
| 직군 영향 | 큼 (직군별 성장 경로·기여 방향이 다름) |
| 채용 담당자가 보는 것 | 현실성 · 회사와의 합치 · 단·중기 액션 구체성 |

## 3.2 이 항목이 묻는 것

입사 후 1년·3년·5년 등 시간 축에서 어떻게 기여할지를 본다. 추상적인 '함께 성장하겠습니다'가 아니라 회사의 로드맵·기술 스택·도전과제에 맞춘 구체적 액션을 답해야 한다.

## 3.3 직군별 작성 전략

- 백엔드: 1년차 기여(서비스 안정성·코드 품질) → 3년차 도전(아키텍처 개선·도메인 책임) → 5년차 영향(팀·조직 기술 표준)
- 프론트엔드: 1년 핵심 화면 임팩트 → 3년 디자인 시스템·성능 표준화 → 5년 프론트엔드 아키텍처 리드
- 인프라/DevOps: 1년 자동화 정착 → 3년 관측성·신뢰성(SLO) → 5년 SRE 문화 정착
- 인프라 시스템 운영: 1년 백업·보안 표준화 → 3년 가상화·자산 자동화 → 5년 ITSM·BCP 책임자
- UI/UX 디자이너: 1년 핵심 플로우 사용성 개선 → 3년 디자인 시스템 운영 → 5년 디자인 리더십

## 3.4 마스터 답변 샘플

▶ 백엔드 (PRAR + 시간축) 샘플  ·  백엔드 3년차 이직 가정  ·  472자

입사 후 1년·3년·5년의 기여 그림을 다음과 같이 그리고 있습니다.

[1년차] 합류 후 첫 3개월은 결제·정산 도메인 코드베이스와 운영 현황을 파악하고, 직전 회사에서 적용한 PRAR 접근으로 가장 시급한 병목 한 가지를 정량 개선하는 것을 목표로 합니다.

[3년차] 결제 정합성·이벤트 기반 정산 영역을 책임 도메인으로 가져가, MSA 전환의 핵심 모듈을 설계·구현하고자 합니다. 직전 회사에서 도입한 Kafka 비동기 처리·Outbox 패턴 경험을 기반으로 정합성 보장 표준을 팀 내에서 수립하는 것이 목표입니다.

[5년차] 결제 도메인 테크 리드로 성장해 신규 글로벌 결제 게이트웨이 통합을 주도하고, 사내 백엔드 표준(코드 컨벤션·아키텍처·관측성)을 정립해 후배가 동일한 시행착오를 반복하지 않도록 돕고 싶습니다.

▶ 프론트엔드 (UX-Driven + 시간축) 샘플  ·  프론트 2년차 이직 가정  ·  415자

입사 후 단계별 기여 계획은 다음과 같습니다.

[1년차] 합류 후 6개월 내 핵심 결제 플로우의 LCP를 측정·개선해 사용자 이탈률을 정량적으로 낮추는 것을 첫 목표로 합니다. 직전 회사에서 LCP 4.2초 → 1.6초, 이탈률 18% → 9% 개선 경험이 있고, 같은 접근을 빠르게 적용할 수 있다고 판단합니다.

[3년차] 디자인 시스템 코드 구현을 책임지고 Storybook·시각 회귀 테스트를 운영해 디자이너-개발 협업의 핸드오프 비용을 줄이는 것이 목표입니다.

[5년차] 프론트엔드 아키텍처(번들·렌더링 전략·상태 관리 표준) 의사결정을 주도하고, 신입·주니어가 일관된 패턴으로 작업할 수 있는 가이드를 운영하고 싶습니다.

## 3.5 흔한 실수

- "열심히 하겠습니다" 류 추상 선언
- 시간축이 모호함 (1년/3년 구분 없음)
- 회사 로드맵 무시한 일반론
- 본인이 이전 회사에서도 안 해본 일을 약속
- "CTO가 되겠습니다" 식 비현실적 목표

## 3.6 AI 프롬프트 핵심 지시

```
[필수 포함]
1. 시간축 명시 (1년차 / 3년차 / 5년차 또는 단·중·장기)

- 본인 입장이 절대적으로 옳았다는 톤 회피
- 기술적 갈등도 OK (예: 아키텍처 선택, 디자인 방향)

[인사이트] (1) 데이터: GA 퍼널에서 카드 정보 입력 단계 이탈이 가장 컸음 (23%). (2) 사용자 인터뷰 8건: '카드번호 16자리를 한 번에 입력하는 게 부담', '실제 카드가 손에 없을 때 정보를 못 찾는다', '에러가 나도 무엇이 틀렸는지 모르겠다'. (3) 휴리스틱 평가: 한 화면에 11개 입력 필드가 동시에 노출.

[솔루션] 3개 솔루션을 우선순위화해 적용. (1) 입력 필드를 3단계 step으로 분리(카드 정보 → 청구 정보 → 확인), (2) 카드번호는 4자리씩 자동 포커스 이동 + 카드사 자동 인식, (3) 실시간 검증과 정확한 에러 메시지(예: '카드 번호 13번째 자리가 누락되었습니다').

[검증] A/B 테스트(2주, n=12,000) 결과 이탈률 32% → 12%, CS 문의 200건/월 → 35건, 결제 완료까지 평균 소요 시간 87초 → 42초. 변경 솔루션을 디자인 시스템 컴포넌트로 추출해 다른 화면에도 재사용 가능하도록 정리했습니다.

[배움] 단일한 큰 변화보다 데이터로 검증된 작은 변화의 누적이 안정적인 임팩트를 만든다는 것을 체득했습니다.

## 7.2 ACHIEVEMENT와 차이

ACHIEVEMENT가 '결과의 크기'를 강조한다면, PROBLEM_SOLVING은 '문제 분석·접근의 깊이'를 강조한다. 작은 임팩트라도 원인 분석이 깊고 재발 방지까지 챙겼다면 좋은 답이 된다. 같은 사례로 ACHIEVEMENT와 PROBLEM_SOLVING 두 항목을 쓸 때는 강조점을 다르게 잡아야 한다.

## 7.3 직군별 작성 전략

- 백엔드: 장애·버그 추적 및 재발 방지. 디버깅 로그 단계 명시
- 프론트엔드: 사용자 보고 이슈 → 재현 → 원인 → 해결 → 모니터링
- 인프라 클라우드: 장애 대응 → 포스트모템 → 자동화 도입
- 인프라 시스템 운영: 사고 대응(랜섬웨어·해킹·데이터 유실) → 격리·복구 → SOP 수립
- UI/UX 디자이너: 사용성 이슈 발견 → 원인 분석 → 디자인 개선 → 재검증

## 7.4 마스터 답변 샘플

▶ 백엔드 (PRAR + 디버깅 흐름) 샘플  ·  프로덕션 메모리 누수 추적  ·  818자

직전 회사에서 프로덕션 서버가 매주 한 번씩 OOM(Out Of Memory)으로 재시작되는 문제를 추적·해결한 경험입니다.

[P · 문제] 매주 토요일 새벽 2시경 결제 API 서버 3대 중 1대가 OOM으로 재시작되었고, 그 시점 약 5분간 결제 실패가 발생했습니다. 운영팀은 '서버 사양을 늘리자'고 제안했지만 근본 원인이 명확하지 않은 채 자원만 추가하는 것은 부담이었습니다.

[R · 원인] 3주에 걸쳐 추적했습니다. (1) JVM 힙 덤프를 매일 분석해 GC 패턴에서 영구히 살아남는 객체가 시간당 약 1.2MB 증가하는 것을 확인. (2) MAT(Memory Analyzer Tool)로 leak suspect 분석 → 특정 캐시 클래스가 무제한 증가. (3) 코드 추적 결과 캐시에 (사용자ID + 요청시각) 키를 사용해 사실상 무한히 키가 늘어나는 구조였고, TTL이 설정되지 않은 상태였습니다. 매주 토요일 새벽에 일괄 작업으로 트래픽이 집중되면서 OOM 임계치에 도달.

[A · 접근] (1) 즉시 캐시 키에서 시각을 제거하고 TTL 30분 설정으로 핫픽스 배포. (2) 캐시 컴포넌트에 max size + TTL 필수 검증 어노테이션을 추가해 같은 실수가 코드 리뷰에서 잡히도록 도입. (3) 메모리 사용량을 Prometheus 메트릭으로 노출하고 임계치 알림 구성.

[R · 결과] 핫픽스 후 OOM 재발생 0건, 평균 힙 사용량 75% → 42%, 결제 실패 0건. 캐시 가이드 문서를 작성해 사내 백엔드 공통 자료로 등록했습니다.

[배움] '서버 사양 증가'는 대증요법일 뿐이며, 정량적 측정으로 근본 원인을 찾아야 한다는 것, 그리고 코드 단의 가드(어노테이션·테스트)로 같은 실수의 재발을 막을 수 있다는 것을 체득했습니다.

▶ 인프라 시스템 운영 (Ops-Result) 샘플  ·  내부망 속도 저하 원인 추적  ·  690자

사내 네트워크가 갑자기 느려져 모든 부서에서 '인터넷이 안 된다'는 문의가 폭주한 사건을 해결한 경험입니다.

[문제] 어느 오후 갑자기 사내 모든 PC에서 인터넷 속도가 1/10 수준으로 떨어졌습니다. 헬프데스크 티켓 1시간 만에 30건 누적, 임원실에서까지 직접 전화가 왔습니다.

[원인 분석] (1) 외부 회선 ISP 측정: 정상. (2) 사내 코어 스위치 트래픽 모니터링: 특정 포트 1개에서 1Gbps 트래픽 발생(정상은 평균 50Mbps). (3) 해당 포트 추적 → 마케팅팀 PC 1대. (4) 그 PC를 격리 분석 → 사용자가 모르는 사이 4K 동영상 200개가 무한 루프로 클라우드에 업로드되는 스크립트가 실행 중. 알고 보니 사용자가 USB에 받은 무료 동영상 도구가 백그라운드 업로더였습니다.

[조치·정책] (1) 해당 PC 즉시 격리·재이미징. (2) 방화벽에 비업무 시간 외 대용량 업로드 Rate Limit 추가. (3) EDR에 비인증 소프트웨어 실행 차단 룰 추가. (4) 전사 보안 교육에 '무료 도구 위험' 사례로 추가.

[결과] 사건 후 30분 내 네트워크 정상화, 이후 6개월간 유사 사고 0건. 본 사고를 분기 보안 보고서의 우수 대응 사례로 등록.

[배움] 사고는 외부가 아닌 내부 사용자 행위에서 더 자주 발생하며, 기술적 차단과 사용자 교육이 같이 가야 재발을 막을 수 있다는 것을 체득했습니다.

## 9.3 모든 직군 공통 전략

- 직무 치명적이지 않은 것 (예: 백엔드 지원자가 '커뮤니케이션' 약점은 위험)
- 현재 진행 중인 개선 노력을 구체적으로
- 개선의 정량 결과 (있다면)

## 9.4 마스터 답변 샘플

▶ 공통 (백엔드 톤) 샘플  ·  거절 어려움 → 우선순위 도구화  ·  540자

저의 약점은 '동료의 요청을 우선순위 없이 받아들이는 경향'이 있다는 점입니다. 직전 회사에서 다른 팀의 요청을 모두 그 자리에서 'OK'라고 답하다 보니, 정작 본인 스프린트 목표가 후순위로 밀리는 일이 분기마다 2~3회 발생했습니다.

자각 후 두 가지를 도입했습니다. (1) 모든 외부 요청을 받으면 24시간 이내 답하는 룰을 만들고, 즉답을 피하기 위해 '내일 답드리겠다'로 우선 응답. (2) 요청을 받은 직후 본인 스프린트 보드에 적어두고, 다음 데일리 스탠드업에서 팀과 우선순위를 합의한 후 회신.

이 두 가지 룰을 적용한 후 분기당 본인 목표 달성률이 70% → 95%로 개선되었고, 동료들의 요청도 더 명확한 답을 받게 되어 만족도 면담에서 긍정적 피드백을 받았습니다. 여전히 본질적으로 'No'를 직접 말하는 것이 편하지는 않지만, 룰을 도구로 보완하는 방식을 통해 약점이 본인과 팀의 성과에 영향을 미치지 않도록 관리하고 있습니다.

## 9.5 흔한 실수

- 강점 위장: "꼼꼼해서 시간이 오래 걸려요"
- 직무 치명적 약점: 개발자가 "논리적 사고가 약해요"
- 개선 노력 없음 ("그게 약점이라는 걸 압니다"로 끝)
- "완벽주의" 같은 클리셰

## 9.6 AI 프롬프트 핵심 지시

```
[필수 포함]
1. 실제 약점 (강점 위장 금지)

3. 현재 진행 중인 개선 노력 (구체적 방법·도구)
4. 개선 결과 (정량 또는 변화 묘사)

[금지]

## 10.2 이 항목이 묻는 것

"자유롭게 본인을 표현하세요", "위 항목 외 자유 기술", "이 자리에 본인을 어필하세요" 등 정형화되지 않은 항목. v1에서는 사용자가 직접 항목 질문 텍스트를 입력하고, AI는 그 질문 텍스트를 핵심 컨텍스트로 활용한다.

## 10.3 작성 전략

- 공고 질문의 핵심 키워드를 가장 먼저 식별
- 질문 키워드 → 직군 권장 구조 자동 매핑 시도 (예: '성장 경험' → ACHIEVEMENT 유사 처리, '비전' → FUTURE_PLAN 유사 처리)
- 적합한 유사 항목이 없으면 자유 작성 + 정량 결과·구체 사례 원칙 유지

   적용 (다른 8개 유형의 가이드 중 하나를 참조 추가)

# 부록 B. 사용자 노출 가이드 카피

UI에서 사용자가 마스터를 새로 만들거나 항목 유형을 고를 때 표시할 가이드 텍스트.

### B.1 항목 유형 선택 시 카드 카피 (한 줄)

- 지원 동기 — 회사와 직무가 본인의 경험과 어떻게 연결되는가
- 입사 후 포부 — 1년·3년·5년 시간축에서 어떻게 기여할지
- 협업 경험 — 다른 사람들과 어떻게 정렬해 일을 진행했는가
- 갈등 해결 경험 — 의견 차이를 어떻게 사실 기반으로 해소했는가
- 성취 경험 — 정량 임팩트와 본인 기여를 강조
- 문제 해결 경험 — 원인 분석과 재발 방지를 강조
- 본인의 강점 — 회사 합치되는 1~2개 강점 + 근거 사례
- 본인의 약점 — 자기 객관화 + 현재 개선 노력
- 기타 — 공고의 자유 질문에 맞춰 작성

### B.2 거짓방지 검토 표시 가이드 (도움말)

AI 초안이 사용자 이력에 없는 표현을 사용한 경우 '검토 표시'가 뜹니다. 다음 두 경우 중 하나로 처리하세요.

- ① 사실에 부합 → 사용자가 직접 [검토 완료] 클릭. 사용자가 작성한 user_edit 영역에 반영됩니다.
- ② 사실에 부합하지 않음 → 해당 표현을 삭제하거나 사용자 이력의 실제 사실로 교체.
