# Spec — feat/5.7-cover-letter-write (AI 자소서 작성)

작성일: 2026-05-21
브랜치: `feat/5.7-cover-letter-write`
직전 베이스: `feat/5.6-cover-letter-master` 머지 후 develop (또는 develop 직접)

## 1. 배경 / 목표

§6 자소서 작성은 **v1 의 핵심 가치 화면**: 사용자가 등록한 채용공고에 대해 항목별로 AI 초안을 생성 → 좌측에 초안 보존 + 우측에서 편집 → 실시간 검증 (글자수 + JD 키워드 매칭) → 저장.

본 spec 은 §6 작성·저장·관리 도메인 전체를 다루며, 동시에 **5.6 마스터 자소서 입력 페이지를 v1 에서 폐기** (사용자 피로감 감소를 위한 pivot).

## 2. 결정 내역 (브레인스토밍 합의)

1. **5.6 마스터 자소서 폐기** — TopNav "이직/취업" 메뉴에서 `/cover-letters/master` 진입 차단. BE controller/service/entity/encryptor 는 그대로 보존 (variant 의 ai_draft/user_edit 암호화에 encryptor 재사용 + v2/v3 "잘된 자소서 저장" 기능에 entity 재활용 가능성).
2. **v1 자소서 입력 소스**: 경력(5.4) + 채용공고(5.5) + **선택적** 기업분석(§9) + 사용자 요청사항 free text. 마스터 답변 연결 제거.
3. **진입 경로**: (A) 공고 카드(5.5)에 "✍ 자소서 작성" 버튼 + (B) `/cover-letters` 자소서 목록 페이지의 "AI 자소서 작성" CTA — 두 경로 모두 지원.
4. **검증 사이드 패널 v1**: 글자수 카운트 + JD 키워드 매칭 (본문 vs `job_posting.keywords` string match). 거짓 방지·맞춤법 v2.
5. **저장 액션**: 임시 저장 (status=DRAFT) + 완료 저장 (status=COMPLETED). PDF/Word 다운로드 v2.
6. **JD 키워드 매칭 trigger**: 사용자 입력 중 실시간 (debounce 500ms). 클라이언트 string match 라 BE 호출 없음.
7. **자소서 목록 그룹화**: 회사별 그룹 + 그룹 내 최근수정순.
8. **§9 기업분석은 별도 phase**: 본 spec 의 "기업분석 연결" select 는 초기엔 빈 placeholder (옵션 없음 → 비활성). §9 spec/plan 후속 머지 시 자동으로 옵션 표시.
9. **AES-256 암호화 재사용**: `cover_letter_variant.ai_draft`, `cover_letter_variant.user_edit` 두 컬럼을 기존 `MasterAnswerEncryptor` 로 동일하게 암호화. 별도 키 분리 안 함 (운영 키 관리 단순화).
10. **OpenAI 호출**: gpt-4o-mini 기본 (5.5 LLM 키워드 추출 모듈과 동일). 환경변수 OPENAI_API_KEY/OPENAI_MODEL 재사용.

## 3. v1 / v1 비범위

### v1

**FE**
- TopNav "이직/취업" 메뉴 갱신: `/cover-letters/master` → `/cover-letters` (목록)
- `/cover-letters/master` 라우트는 404 (또는 `/cover-letters` 로 redirect)
- `/cover-letters` — 자소서 목록 페이지 (회사별 그룹 + 최근수정순)
- `/cover-letters/variants/new?postingId={id}&itemType={enum}` — §6 작성 화면 (좌·우 분할 + 사이드 패널)
- `/cover-letters/variants/{id}` — 편집/조회 (작성 화면과 동일 컴포넌트 재사용, mode=edit)
- 5.5 공고 카드에 "✍ 자소서 작성" 버튼 추가 → 항목 선택 모달 → `/variants/new`
- Sidenav 메뉴: "이직/취업" 카테고리 → "자소서" (`/cover-letters`) + "경력기술서" (`/me/career` cross-link)

**BE**
- `cover_letter_variant` 엔티티 (스키마 이미 V1 migration 에 정의됨)
- `MasterAnswerEncryptor` 재사용 — variant.ai_draft, variant.user_edit AES-256
- CRUD controller `/api/v1/cover-letter-variants/*`
- AI 초안 생성 service — OpenAI 호출 + 프롬프트 조립 (직군 × 항목 유형 분기)
- JD 키워드 매칭 endpoint — 본문 vs job_posting.keywords (BE-side 도 제공, FE-side debounced match 와 별도 — 저장 시 validation_json 채우는 용도)

### v1 비범위 (v2/v3)

- 거짓 방지 검토 (이력 vs 자소서 LLM 대조)
- 맞춤법 점검 (LLM 또는 외부 API)
- PDF/Word 다운로드
- 휴지통 (soft delete UI — schema 에 deleted_at 컬럼 이미 있음, UI 만 v2)
- "잘된 자소서" 즐겨찾기/베이스 활용 (현재 ai_draft + user_edit 외 별도 "saved" 플래그 필요)
- 항목별 PRAR 4단 자동 분리
- 동일 공고의 여러 항목 한 화면 작성 (현재는 한 화면 = 한 항목)
- 기업분석(§9) 연결 — 별도 phase 의 spec/plan 으로 진행

## 5. API 설계

| Method | Path | 요청 | 응답 | 비고 |
|---|---|---|---|---|
| GET | `/api/v1/cover-letter-variants/grouped` | — | `VariantListGroupedResponse[]` | 회사별 그룹·최근수정순 |
| GET | `/api/v1/cover-letter-variants/{id}` | — | `VariantResponse` | ai_draft, user_edit 복호화 |
| POST | `/api/v1/cover-letter-variants` | `VariantCreateRequest` | 201 `VariantResponse` | AI 초안 생성 + DRAFT 저장 |
| PATCH | `/api/v1/cover-letter-variants/{id}` | `VariantPatchRequest` | `VariantResponse` | userEdit, status, userRequest 부분 갱신 |
| DELETE | `/api/v1/cover-letter-variants/{id}` | — | 204 | hard delete (v2 에 soft delete 추가) |
| GET | `/api/v1/cover-letter-variants/{id}/validation` | — | `ValidationResponse` | JD 키워드 매칭·글자수 (서버 검증) |

`VariantCreateRequest`:
```ts
{
  postingId: number;
  itemType: ItemType;             // 9 enum
  charLimit: number;              // > 0
  userRequest?: string;           // 사용자 요청사항
  analysisId?: number;            // Phase C 머지 후 사용
}
```

`VariantPatchRequest`:
```ts
{
  userEdit?: string;
  status?: "DRAFT" | "COMPLETED";
  userRequest?: string;
}
```

`VariantResponse`:
```ts
{
  id: number;
  postingId: number;
  postingCompany: string;          // join 으로 표시
  postingTitle: string;
  analysisId: number | null;
  itemType: ItemType;
  question: string;                // 항목별 질문 (item-type-questions 매핑)
  charLimit: number;
  aiDraft: string | null;          // 복호화 평문
  userEdit: string | null;         // 복호화 평문
  userRequest: string | null;
  validationJson: object | null;
  status: "DRAFT" | "COMPLETED";
  aiModel: string | null;
  aiTokensUsed: number | null;
  createdAt: string;
  updatedAt: string;
}
```

`VariantListGroupedResponse`:
```ts
{
  posting: { id, company, title };
  variants: VariantSummaryResponse[];
}
```

`VariantSummaryResponse` (목록·그룹 용):
```ts
{ id, itemType, charCount, status, updatedAt }
```

`ValidationResponse`:
```ts
{
  charCount: number;
  charLimitOk: boolean;
  matchedKeywords: string[];       // posting.keywords ∩ edit
  matchCount: number;
  matchOk: boolean;                // matchCount >= 3 (기준)
}
```

## 7. 검증 로직

### FE 실시간 (debounce 500ms)
```ts
// write-validation-panel.tsx
const charCount = userEdit.length;
const charLimitOk = charCount <= charLimit && charCount >= charLimit * 0.9;
const matched = postingKeywords.filter(kw => userEdit.includes(kw));
const matchOk = matched.length >= 3;
```

### BE 저장 시 validation_json
PATCH /variants/{id} 의 응답에 validation_json 갱신 (string match + 글자수). 거짓방지·맞춤법은 v1 비범위라 null 유지.

좌(초안)는 항상 읽기전용 회색 톤. 우(편집)는 흰 톤 + 색깔 강조 (현재 글자수 색상: 90~100% green, 100~110% yellow, 110%+ red).

새 작성 모드 = 입력 패널 채워진 상태로 진입, "AI 초안 생성" 안 누르면 좌·우 비어있음. 생성 후엔 입력 패널 readonly (수정 시 새 variant 만들어야 함 → 또는 "초안 재생성" 액션, v2).

편집 모드 = 입력 패널 readonly (postingId 등 변경 불가), 좌(저장된 ai_draft) + 우(저장된 user_edit) 즉시 표시.

### 9.3 공고 카드 (5.5)

PostingCard 의 actions row 우측에 "✍ 자소서 작성" 버튼 추가 → onClick → 항목 선택 모달.

## 10. 미해결 / 위험

1. **OpenAI 비용**: 한 자소서 생성 ~$0.01 (gpt-4o-mini 가격 기준). 사용자당 10~30회 가정 시 월 ~$0.3. 모니터링: ai_tokens_used 컬럼 로깅.
2. **OpenAI 응답 품질 변동성**: 같은 입력으로도 응답이 다름. v1 은 사용자가 우측에서 편집 가능하므로 허용. 품질 개선은 v2 의 프롬프트 튜닝.
3. **글자수 카운트 정확성**: HTML 텍스트의 공백·줄바꿈을 한국어 글자수로 어떻게 셀지. v1 은 단순 .length (모든 문자 1개로 카운트). 더 정교한 카운트는 v2.
4. **§9 와의 통합 시점**: Phase C (§9) 미머지 동안 "기업분석 연결" select 는 빈 옵션. UX 적으로 비활성된 selectbox 가 사용자에게 혼란일 수 있음 → "곧 추가 예정" 한 마디 helper text 표시 (v1 spec 에 포함).
5. **5.6 잔재 코드 보존의 향후 부담**: dead code 가 의도적으로 남아있어 향후 v2/v3 에서 재활성화·재구성 시 헷갈릴 수 있음. 잔재 파일에 `// DEPRECATED: 5.6 v1 사용 안 함. v2 의 "잘된 자소서 저장" 에서 재활용 후보` 주석 1줄 추가.
