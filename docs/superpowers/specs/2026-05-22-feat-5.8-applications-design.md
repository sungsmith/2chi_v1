**작성일:** 2026-05-22
**대상 섹션:** wireframes §7 (캘린더) + §8 (지원 대시보드)
**브랜치:** `feat/5.8-applications`

## 1. Goal & Context

**한 줄 요약.** 사용자가 지원한 공고의 전형 진행과 일정을 한 화면에 모으는 "지원 현황" 도메인을 v1 에 도입한다. 시각화는 캘린더(§7) 와 대시보드 목록(§8) 두 가지 뷰로 제공한다.

**왜 필요한가.** v1 에서 사용자는 §5 공고 등록 + §6 자소서 작성까지는 가능하지만, "어디까지 지원했고 다음 일정이 언제인지" 가 빠져 있다. §3 대시보드의 "다가오는 일정" 패널도 mock 으로만 있고 "캘린더 보기" 버튼이 disabled 다. 본 작업으로 그 빈자리를 채운다.

**범위 결정 요약 (브레인스토밍 단계 합의):**
- §7 캘린더 + §8 대시보드 묶음 (한 PR)
- `Application` 은 `JobPosting` 에 종속 (1 공고 = 0 또는 1 지원, UNIQUE) — §5 공고 카드의 ✅지원함 버튼이 유일한 생성 경로
- `Event` 는 `Application` 의 1:N 자식 (전형 단계별 일정 N 개 + 사용자 임의 일정)
- 캘린더: 월 보기 only, 클릭 = 편집 모달 (좌클릭 팝오버/우클릭 컨텍스트 메뉴/주·일 보기/드래그&드롭/커스터마이징/알림 모두 v2)
- 대시보드: 필터(전형 chip) + 정렬 + 행 클릭 = 편집 모달 (인라인 편집/핀/페이징 모두 v2)
- §3 대시보드의 "다가오는 일정" 패널을 본 PR 에서 실데이터 연결

## 2. Architecture & Data Model (BE)

### 2.1 Entity

```
Application
  id (PK, BIGSERIAL)
  user_id (FK users.id, NOT NULL)
  posting_id (FK job_postings.id, NOT NULL)
  company VARCHAR(120) NOT NULL    -- posting 에서 복사한 스냅샷
  role VARCHAR(120) NOT NULL       -- posting 에서 복사한 스냅샷
  current_stage VARCHAR(30) NOT NULL  -- Stage enum
  current_result VARCHAR(20) NOT NULL -- Result enum, default IN_PROGRESS
  memo TEXT
  created_at TIMESTAMP NOT NULL
  updated_at TIMESTAMP NOT NULL

Event
  id (PK, BIGSERIAL)
  application_id (FK applications.id ON DELETE CASCADE, NOT NULL)
  type VARCHAR(30) NOT NULL        -- EventType enum
  event_date DATE NOT NULL
  event_time TIME                  -- nullable (all-day 일정 허용)
  memo TEXT
  created_at TIMESTAMP NOT NULL
  updated_at TIMESTAMP NOT NULL
```

### 2.2 Enum

**Stage** (application.current_stage)
- `DOC_SUBMITTED` — 서류 제출
- `CODING_TEST` — 코딩 테스트
- `FIRST_INTERVIEW` — 1차 면접
- `SECOND_INTERVIEW` — 2차 면접
- `EXEC_INTERVIEW` — 임원 면접
- `NEGOTIATION` — 처우 협의
- `PASSED` — 합격
- `FAILED` — 불합격

**Result** (application.current_result)
- `IN_PROGRESS` (default) — 진행 중
- `PASSED` — 합격
- `FAILED` — 불합격
- `WITHDRAWN` — 포기

**EventType** (event.type) — Stage 와 의미 묶음 + 추가 종류
- `DOC_DEADLINE` — 서류 마감 (캘린더에 표시되는 마감 알람)
- `CODING_TEST` — 코딩 테스트
- `FIRST_INTERVIEW` — 1차 면접
- `SECOND_INTERVIEW` — 2차 면접
- `EXEC_INTERVIEW` — 임원 면접
- `NEGOTIATION` — 처우 협의
- `PASSED` — 합격 통보
- `FAILED` — 불합격 통보
- `ETC` — 사용자 임의 일정 (예: "OO 회사 커피챗")

### 2.3 Migration V3

V1 schema 가 이미 `application` + `application_history` 테이블을 사전 정의해두었으나 (DOCUMENT/INTERVIEW_1 등 다른 stage naming + dates_json JSONB 모델), brainstorming 단계에서 Application + Event 1:N 모델과 새 stage naming 으로 결정했으므로 V1 의 두 테이블을 제거하고 새 schema 로 교체한다. FK 참조는 V1 의 실제 테이블명 `app_user` / `job_posting` (singular) 을 사용한다.

```sql
DROP TABLE IF EXISTS application_history;
DROP TABLE IF EXISTS application;

CREATE TABLE applications (
    id BIGSERIAL PRIMARY KEY,
    user_id BIGINT NOT NULL REFERENCES app_user(id) ON DELETE CASCADE,
    posting_id BIGINT NOT NULL REFERENCES job_posting(id) ON DELETE CASCADE,
    company VARCHAR(120) NOT NULL,
    role VARCHAR(120) NOT NULL,
    current_stage VARCHAR(30) NOT NULL,
    current_result VARCHAR(20) NOT NULL,
    memo TEXT,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE UNIQUE INDEX uq_application_user_posting
    ON applications (user_id, posting_id);

CREATE TABLE events (
    id BIGSERIAL PRIMARY KEY,
    application_id BIGINT NOT NULL REFERENCES applications(id) ON DELETE CASCADE,
    type VARCHAR(30) NOT NULL,
    event_date DATE NOT NULL,
    event_time TIME,
    memo TEXT,
    created_at TIMESTAMP NOT NULL,
    updated_at TIMESTAMP NOT NULL
);

CREATE INDEX ix_event_application ON events (application_id);
CREATE INDEX ix_event_date ON events (event_date);
```

### 2.4 자동 이벤트 생성 정책

`POST /applications {postingId}` 호출 시:
1. `(userId, postingId)` UNIQUE 검사
2. JobPosting 조회 → company/role 스냅샷 복사
3. Application 생성 (`current_stage=DOC_SUBMITTED`, `current_result=IN_PROGRESS`)
4. `posting.deadline` 이 not null 이면 `Event(type=DOC_DEADLINE, event_date=posting.deadline)` 1 개 자동 생성
5. 응답에 application + 생성된 event 1 개 포함

### 2.5 관계

- `JobPosting.deadline` 필드는 그대로 둔다 — application 생성 시 1 회 복사만 한다. 이후 posting.deadline 변경은 application 의 event 에 영향 없음 (사용자가 수동 갱신).
- `CoverLetterVariant.postingId` 와의 매칭은 BE 조인으로 처리 (`variantsCount = COUNT(variants WHERE posting_id = application.posting_id)`). DB schema 변경 없음.

## 3. API Design

### 3.1 Application CRUD

| Method | Path | Body | Response | Notes |
|---|---|---|---|---|
| POST | `/api/v1/applications` | `{postingId}` | `ApplicationResponse` | 409 if `(userId, postingId)` 중복 |
| GET | `/api/v1/applications` | query: `stage?, result?, sort?` | `ApplicationSummaryResponse[]` | 페이징 없음 |
| GET | `/api/v1/applications/{id}` | — | `ApplicationResponse` (events 포함) | 다른 user 자원 404 |
| PATCH | `/api/v1/applications/{id}` | `{currentStage?, currentResult?, memo?, company?, role?}` | `ApplicationResponse` | |
| DELETE | `/api/v1/applications/{id}` | — | 204 | events CASCADE |

`sort` 값: `updatedAt` (default), `deadline` — `deadline` 은 application 의 `DOC_DEADLINE` event 의 `event_date` 기준 ASC NULLS LAST.

### 3.2 Event CRUD

| Method | Path | Body | Response | Notes |
|---|---|---|---|---|
| POST | `/api/v1/applications/{applicationId}/events` | `{type, eventDate, eventTime?, memo?}` | `EventResponse` | application 의 user_id 검증 |
| PATCH | `/api/v1/events/{id}` | `{type?, eventDate?, eventTime?, memo?}` | `EventResponse` | event 의 application 의 user_id 검증 |
| DELETE | `/api/v1/events/{id}` | — | 204 | |

### 3.3 Event 통합 조회 (cross-application)

| Method | Path | Query | Response |
|---|---|---|---|
| GET | `/api/v1/events` | `from=YYYY-MM-DD&to=YYYY-MM-DD` | `EventListItemResponse[]` |

응답 정렬: `event_date ASC, event_time ASC NULLS LAST`.

용도: 캘린더 월별 표시 (`from=monthStart, to=monthEnd`), 대시보드 §3 패널 (`from=today, to=today+7`).

### 3.4 DTO

```java
// 요청
record ApplicationCreateRequest(@NotNull Long postingId) {}
record ApplicationPatchRequest(
    Stage currentStage, Result currentResult,
    String memo, String company, String role
) {}

    @NotNull EventType type,
    @NotNull LocalDate eventDate,
    LocalTime eventTime,
    String memo
) {}
record EventPatchRequest(EventType type, LocalDate eventDate, LocalTime eventTime, String memo) {}

// 응답
record ApplicationResponse(
    Long id, Long postingId, String company, String role,
    Stage currentStage, Result currentResult, String memo,
    Integer variantsCount, List<EventResponse> events,
    Instant createdAt, Instant updatedAt
) {}
record ApplicationSummaryResponse(
    Long id, Long postingId, String company, String role,
    Stage currentStage, Result currentResult,
    Integer variantsCount, EventResponse nextEvent,
    Instant updatedAt
) {}
record EventResponse(
    Long id, Long applicationId, EventType type,
    LocalDate eventDate, LocalTime eventTime, String memo,
    Instant createdAt, Instant updatedAt
) {}
record EventListItemResponse(
    Long id, EventType type,
    LocalDate eventDate, LocalTime eventTime, String memo,
    Long applicationId, String company, String role
) {}
```

`nextEvent` = `events` 중 `event_date >= today` 인 가장 가까운 1 개. 없으면 null.

### 3.5 Error codes (신규)

```java
APPLICATION_ALREADY_EXISTS(HttpStatus.CONFLICT, "이미 지원한 공고입니다."),
APPLICATION_NOT_FOUND(HttpStatus.NOT_FOUND, "지원을 찾을 수 없어요."),
EVENT_NOT_FOUND(HttpStatus.NOT_FOUND, "일정을 찾을 수 없어요."),
```

### 3.6 보안 패턴 (5.9 와 동일)

- 모든 endpoint 는 `principal.userId()` 로 소유 검증.
- `findByIdAndUserId(id, userId)` 또는 service 레이어의 `findOwned(userId, id)` 패턴.
- Event endpoint 는 event → application → user_id 체인 검증.
- 다른 user 자원 접근 시 403 대신 404 반환 (정보 노출 방지).

### 4.3 §7 캘린더 — 컴포넌트 트리

```
/applications/calendar/page.tsx (server)
  └─ <CalendarContent /> (client)
```

**Month 계산 로직:**
- `useSearchParams` 로 `month=YYYY-MM` 파싱. 없으면 today.
- 그리드 시작 = `monthStart.startOf("week")` (일요일 기준)
- 그리드 끝 = `monthEnd.endOf("week")`
- 6 주 × 7 일 = 42 셀 고정 (월 길이 무관)

**EventCreateModal 필드:**
- Application select dropdown — 현재 user 의 모든 application 표시 (`PASSED`/`FAILED` 도 추가 일정 가능: 합격 통보 후 처우 협의 일정 등). 빈 경우 select 대신 "지원이 없어요. 공고 카드에서 ✅지원함 을 먼저 눌러주세요." 안내 + 모달 즉시 닫기 버튼.
- Type select (9 종, `EVENT_TYPE_LABEL`)
- Date input (default = 모달 진입 일자, 또는 today)
- Time input (optional, all-day 허용)
- Memo textarea

### 4.4 ApplicationEditModal — 공유

```
<ApplicationEditModal application={...} onClose>
  ├─ 회사 input (편집 가능)
  ├─ 직무 input (편집 가능)
  ├─ 진행 단계 select (8 가지 Stage)
  ├─ 결과 select (4 가지 Result)
  ├─ 메모 textarea
  ├─ <EventsList>           — 이 application 의 events 표시
  │    └─ <EventRow>        — type / date / time / memo + 편집/삭제 버튼
  ├─ + 일정 추가 (인라인 form)
  ├─ [삭제] (지원 자체 삭제 — confirm)
  ├─ [취소] [저장]
```

**저장 시 단일 트랜잭션 아님 — 분리 호출:**
- application 자체: `PATCH /applications/{id}`
- 신규 event: `POST /applications/{id}/events`
- 기존 event 수정: `PATCH /events/{id}`
- event 삭제: `DELETE /events/{id}`

각 호출은 개별 처리. 실패 시 partial save 허용 (사용자가 다시 시도). v2 에서 optimistic locking 또는 server-side 일괄 endpoint 고려.

**Stage vs EventType 분리 이유:**
- `Stage` = application 의 현재 진행 위치 (사용자가 직접 입력한 "지금 어디까지 왔는가"). `DOC_SUBMITTED` 부터 시작.
- `EventType` = 일정의 종류 (캘린더에 표시되는 미래/과거 시점). `DOC_DEADLINE` (제출 마감일) 이 있고, 나머지는 Stage 의 8 개와 같은 시간상 상태를 그 시점의 일정으로 표현한 것 + `ETC` (사용자 임의).
- 즉 둘은 의미상 겹치지만 시간축이 다르다: Stage 는 "지금", EventType 은 "그 날짜에 일어나는 일".

**`lib/api/application.ts`** (신규) — 9 개 함수
```ts
createApplication(req: {postingId: number}): Promise<Application>
fetchApplications(filter?: {stage?, result?, sort?}): Promise<ApplicationSummary[]>
fetchApplication(id: number): Promise<Application>
patchApplication(id: number, req: ApplicationPatch): Promise<Application>
deleteApplication(id: number): Promise<void>

createEvent(applicationId: number, req: EventCreate): Promise<Event>
patchEvent(id: number, req: EventPatch): Promise<Event>
deleteEvent(id: number): Promise<void>

fetchEvents(from: string, to: string): Promise<EventListItem[]>
```

## 5. State, Data Flow, Error Handling

### 5.1 State pattern (5.9 와 동일)

- 페이지: server component `page.tsx` → client `*Content` 컴포넌트로 위임
- 모달 state: parent `*Content` 의 `useState<{ open: boolean; target?: T }>`
- URL 동기화: `useSearchParams` 로 읽기, `router.replace(?...)` 로 쓰기
- 데이터 fetch: `useEffect(() => { fetchX().then(setX).catch(setErr); }, [deps])`
  - `react-hooks/set-state-in-effect` lint 회피 패턴 (5.9 검증됨)

### 5.2 Data flow — 캘린더

```
1. /applications/calendar?month=2026-05 진입
2. CalendarContent mount → month parse → fetchEvents(monthStart, monthEnd) + fetchApplications()
   (둘 다 Promise.all 병렬)
3. event chip 클릭 → EventEditModal open(target=event)
4. modal save → patchEvent(id, body) → events refetch (같은 month)
5. modal delete → deleteEvent(id) → events refetch
6. + 일정 추가 클릭 → EventCreateModal open
7. modal create → createEvent(applicationId, body) → events refetch
8. 이전/다음 월 → router.replace(?month=...) → useEffect 재실행
```

### 5.3 Data flow — 대시보드

```
1. /applications?stage=DOC_SUBMITTED&result=IN_PROGRESS&sort=updatedAt 진입
2. ApplicationsContent mount → fetchApplications(filter) → list
3. 필터 chip 클릭 → router.replace(?stage=...) → refetch
4. 행 클릭 → fetchApplication(id) → ApplicationEditModal open
5. modal 내 부분 save 4 종:
   - application 자체: patchApplication
   - 신규 event: createEvent
   - 기존 event 수정: patchEvent
   - event 삭제: deleteEvent
6. modal close → list refetch
```

### 5.4 Data flow — 공고 카드 + 대시보드 panel

**공고 카드:**
```
1. PostingsContent mount → fetchPostings() + fetchApplications() 병렬
2. posting → application map 생성 (postingId → applicationId)
3. 카드에 applicationId prop 전달
4. ✅지원함 클릭 → createApplication(postingId) → toast + 카드 상태 갱신
5. ✓ 지원 중 클릭 → router.push("/applications")
```

**§3 대시보드 panel:**
```
1. dashboard-content mount → fetchEvents(today, today+7) 1 회
2. 응답을 ScheduleItem 형태로 변환
3. "캘린더 보기" → router.push("/applications/calendar")
```

### 5.5 Error handling

| 상황 | 응답 | UI 처리 |
|---|---|---|
| 409 `APPLICATION_ALREADY_EXISTS` | posting-card 클릭 시 | toast "이미 지원한 공고입니다" + 카드 상태 즉시 갱신 |
| 404 `APPLICATION_NOT_FOUND` | edit modal 열 때 | 모달 내 빨간 알림 + 닫기 후 list refetch |
| 404 `EVENT_NOT_FOUND` | event 편집/삭제 시 | 모달 내 빨간 알림 + 닫기 후 events refetch |
| 5xx / 네트워크 | 모든 호출 | "잠시 후 다시 시도해 주세요" generic |
| Validation | POST/PATCH body | field-level 빨간 메시지 (기존 form 패턴) |

`role="alert"` 빨간 박스 + 16/13 px 사이즈, `var(--color-semantic-error*)` 토큰 (5.9 와 동일).

`http.ts` 의 problem-detail → message 변환 reuse (이미 ErrorCode 들 매핑).
