## 1. 범위

**포함:**
- `POST /api/v1/auth/login` — 이메일·비번 검증, 5회 실패 시 10분 잠금, Access JWT(30분, JSON 응답) + Refresh(opaque, HttpOnly Cookie + Redis 저장 TTL 14d)
- `POST /api/v1/auth/refresh` — 쿠키 검증 → Redis 매칭 → rotation
- `POST /api/v1/auth/logout` — Redis 에서 refresh 삭제 + Set-Cookie Max-Age=0
- `GET /api/v1/users/me` — JWT 필요. 현재 로그인 사용자 정보 반환 (JWT 동작 검증용)
- JWT 필터 + Spring Security 통합
- 프론트 `AuthContext` + silent refresh fetch wrapper + `/login` 실제 폼

**제외 (다음 브랜치):**
- 소셜 OAuth — `feat/5.1-auth-social`
- 모든 디바이스 로그아웃 / 활성 세션 목록 — 마이페이지
- AuditLog 기록 (LOGIN_SUCCESS / LOGIN_FAIL) — 추후 통합

**스택 추가:**
- `org.springframework.boot:spring-boot-starter-data-redis`
- `io.jsonwebtoken:jjwt-api / jjwt-impl / jjwt-jackson` (0.12.x)

## 2. 전체 흐름

```
[로그인]
  /login → 이메일·비번 → POST /api/v1/auth/login
    → 200 { accessToken, user: {userId, email, nickname} }
    → Set-Cookie: refresh_token=…; HttpOnly; Secure; SameSite=Strict;
                  Path=/api/v1/auth; Max-Age=1209600
    → AuthContext.setAccessToken + setUser
    → router.push('/')

[보호된 요청]
  http() wrapper → Authorization: Bearer <access>
  401 → /refresh 시도 → 새 access → 원래 요청 재시도

[페이지 새로고침]
  AuthProvider 마운트 → /refresh 호출 (브라우저가 쿠키 자동 첨부)
  성공 → access 복원
  실패 → 비로그인 상태

[로그아웃]
  POST /api/v1/auth/logout → Redis 삭제 + Cookie Max-Age=0
  AuthContext clear
```

### 3.2 API 계약

#### POST `/api/v1/auth/login`

Request:
```json
{ "email": "user@example.com", "password": "..." }
```

Response 200:
```json
{
  "accessToken": "eyJhbGciOiJIUzI1NiJ9...",
  "user": { "userId": 1, "email": "...", "nickname": "..." }
}
```

Set-Cookie:
```
refresh_token=<opaque>; HttpOnly; Secure; SameSite=Strict;
Path=/api/v1/auth; Max-Age=1209600
```
(`Secure` 플래그는 `app.refresh.cookie-secure` 로 환경별 분기)

Errors:
| HTTP | code | 상황 |
|---|---|---|
| 401 | INVALID_CREDENTIALS | 이메일 없음 또는 비번 불일치 |
| 423 | ACCOUNT_LOCKED | locked_until > now. body 에 `retryAfterSeconds` 포함 |
| 400 | VALIDATION_FAILED | 입력 형식 오류 (이메일 빈 값 등) |

#### POST `/api/v1/auth/refresh`

Request: body 없음 (HttpOnly cookie 자동 첨부)

Response 200: `{ accessToken, user }` + 새 Set-Cookie (rotation)

Errors:
| HTTP | code | 상황 |
|---|---|---|
| 401 | INVALID_REFRESH_TOKEN | 쿠키 없음 / Redis 매칭 실패 / 만료 |

#### POST `/api/v1/auth/logout`

Request: body 없음

Response 204 + Set-Cookie Max-Age=0. Redis 에서 해당 refresh 삭제 (있으면).

#### GET `/api/v1/users/me`

Header: `Authorization: Bearer <access>`

Response 200: `{ userId, email, nickname, role }`

Errors:
| HTTP | code | 상황 |
|---|---|---|
| 401 | UNAUTHENTICATED | 토큰 없음/위조/만료 |

### 3.3 JWT

`application.yml`:
```yaml
app:
  jwt:
    secret: ${JWT_SECRET:dev-only-change-me-min-32-bytes-please-1234}
    access-token-ttl: PT30M
  refresh:
    cookie-name: refresh_token
    cookie-secure: ${COOKIE_SECURE:false}
    cookie-path: /api/v1/auth
    ttl: P14D
```

Access JWT (HS256) claims:
```
{ "sub": "<userId>", "email", "nickname", "role", "iat", "exp" }
```

Refresh:
- 값: 32 bytes `SecureRandom` → Base64Url
- Redis key: `refresh:<sha256(token)>` (token 평문은 저장하지 않음)
- Value: `userId` (확장 시 lastUsedAt 등 추가)
- TTL: 14일
- Rotation: refresh 사용 시 기존 키 삭제 + 새 토큰 발급/저장

### 3.4 로그인 비즈니스 로직

`LoginService.login(email, rawPassword)`:

1. `userRepository.findByEmailAndDeletedAtIsNull(email)` — Optional
   - empty → throw `BusinessException(INVALID_CREDENTIALS)`
2. `user.isLocked(Instant.now())` — locked_until > now ?
   - true → throw `BusinessException(ACCOUNT_LOCKED, retryAfterSeconds)`
3. `passwordEncoder.matches(rawPassword, user.passwordHash)` — false
   - `user.recordLoginFailure(now)` — failed_login_count +1; >=5 면 locked_until = now+10m
   - save → throw `BusinessException(INVALID_CREDENTIALS)`
4. true (성공)
   - `user.recordLoginSuccess(now)` — failed_login_count=0, locked_until=null, last_login_at=now
   - save
5. JWT 발급, Refresh 발급/Redis 저장, 응답 객체 + 쿠키 반환 책임은 컨트롤러

### 3.5 Spring Security 변경

```
filterChain:
  permitAll: /api/v1/auth/login, /signup, /refresh, /logout, /error, /api/v1/auth/**
  authenticated: 그 외
  JwtAuthenticationFilter: UsernamePasswordAuthenticationFilter 이전 등록
```

`.env.example` 추가:
```

COOKIE_SECURE=false
```

### 4.2 AuthContext

```ts
type AuthUser = { userId: number; email: string; nickname: string };

type AuthContextValue = {
  user: AuthUser | null;
  initialized: boolean;
  login: (email: string, password: string) => Promise<void>;
  logout: () => Promise<void>;
};
```

`AuthProvider` 가 마운트 시 1회 `POST /refresh` 호출:
- 200 → `setAccessToken(token)` + setUser + initialized=true
- 401 → setUser(null) + initialized=true

Access token 은 React state 가 아닌 `lib/api/http.ts` 의 **모듈 변수**에 보관 (`getAccessToken()` / `setAccessToken()`). state 인 것은 user 정보뿐.

### 4.3 lib/api/http.ts (fetch wrapper)

```ts
let accessToken: string | null = null;
let refreshing: Promise<void> | null = null;

export function setAccessToken(t: string | null) { accessToken = t; }
export function getAccessToken() { return accessToken; }

export async function http(input, init: RequestInit = {}) {
  const headers = new Headers(init.headers);
  if (accessToken) headers.set("Authorization", `Bearer ${accessToken}`);
  const res = await fetch(input, { ...init, headers, credentials: "include" });
  if (res.status !== 401) return res;

  // silent refresh
  if (!refreshing) refreshing = refresh().finally(() => { refreshing = null; });
  await refreshing;
  if (!accessToken) return res;

  const retryHeaders = new Headers(init.headers);
  retryHeaders.set("Authorization", `Bearer ${accessToken}`);
  return fetch(input, { ...init, headers: retryHeaders, credentials: "include" });
}

async function refresh() {
  const res = await fetch(`${API_BASE}/api/v1/auth/refresh`, {
    method: "POST", credentials: "include",
  });
  if (!res.ok) { setAccessToken(null); return; }
  const body = await res.json();
  setAccessToken(body.accessToken);
}
```

### 4.4 lib/api/auth.ts 추가

### 4.5 /login 페이지

기존 placeholder 제거 → `LoginForm`:
- 이메일·비번 입력 (TextField 재사용)
- [로그인] 버튼
- 에러 매핑:
  - 401 INVALID_CREDENTIALS → 상단 알림 "이메일 또는 비밀번호가 올바르지 않습니다."
  - 423 ACCOUNT_LOCKED → 상단 알림 "계정이 잠겼습니다. {N}분 후 다시 시도해주세요." (N = ceil(retryAfterSeconds/60))
  - 400 → 상단 알림 (필드별로 분기는 불필요)
  - 5xx → "잠시 후 다시 시도해주세요."
- 성공 → `useAuth().login()` → AuthContext 갱신 → `router.push('/')`
- 하단: "회원이 아니신가요? `/signup`" 링크

| 4회 실패 후 정답 | 카운트 0, last_login_at 갱신 |

`RefreshIntegrationTest`:
| 케이스 | 검증 |
|---|---|
| 정상 refresh | 200, 새 accessToken, 새 Set-Cookie, Redis 의 이전 키 삭제 + 새 키 존재 (rotation) |
| 쿠키 없음 | 401 INVALID_REFRESH_TOKEN |
| 위조 토큰 | 401 |
| logout 후 refresh | 401 |

`LogoutIntegrationTest`:
| 케이스 | 검증 |
|---|---|
| 정상 logout | 204, Set-Cookie Max-Age=0, Redis 에서 삭제 |
| 쿠키 없이 logout | 204 (멱등) |

`MeIntegrationTest`:
| 케이스 | 검증 |
|---|---|
| 유효 Access | 200, {userId, email, nickname, role} |
| Access 없음 | 401 |
| 만료/위조 Access | 401 |

**테스트 인프라:**
- Redis 컨테이너 실사용. `@BeforeEach` 에서 `redisConnectionFactory.connection.flushDb()`.
- 각 테스트에서 다른 이메일 사용 + `@Transactional` 롤백 (failed_login_count 검증 케이스는 별도 정리 필요 시 `@Sql` 또는 `@AfterEach` 사용).
