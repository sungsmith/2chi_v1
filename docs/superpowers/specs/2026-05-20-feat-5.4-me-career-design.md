## 2. 데이터 모델

### 2.1 Career entity

```kotlin
// Java record-like 표기 (Spring Boot + JPA)
@Entity @Table(name = "careers")
class Career {
  @Id @GeneratedValue Long id;
  Long userId;                          // FK to users, NOT NULL
  String companyName;                   // NOT NULL, max 100
  String position;                      // NOT NULL, max 100 (e.g. "백엔드 개발자")
  LocalDate startDate;                  // NOT NULL
  LocalDate endDate;                    // nullable (재직중일 때 null)
  Integer displayOrder;                 // NOT NULL, default 0 (큰 값이 위)
  Instant createdAt;
  Instant updatedAt;
}
```

인덱스: `(userId, displayOrder DESC)` for 정렬 조회.

`endDate` IS NULL → "재직중" UI 라벨.

### 2.2 Project entity

```kotlin
@Entity @Table(name = "projects")
class Project {
  @Id @GeneratedValue Long id;
  Long careerId;                        // FK to careers, NOT NULL
  String title;                         // NOT NULL, max 200
  LocalDate startDate;                  // NOT NULL
  LocalDate endDate;                    // nullable
  Integer contributionPercent;          // 0~100, nullable (선택)

  // PRAR — 모두 nullable (자동저장으로 부분 채우기 허용)
  String prarProblem;                   // text, max 1000
  String prarRootCause;                 // text, max 1000
  String prarApproach;                  // text, max 1000
  String prarResult;                    // text, max 1000

  // JSON 컬럼 (PostgreSQL jsonb)
  String techTags;                      // JSON array of { label, tone? }
  String metrics;                       // JSON array of { k, before?, after?, delta?, dir? }

  Integer displayOrder;                 // NOT NULL, default 0
  Instant createdAt;
  Instant updatedAt;
}
```

인덱스: `(careerId, displayOrder DESC)`.

**JSON 컬럼 선택 이유**:
- techTags/metrics 는 검색·조회 대상 아님 (사용자별 자유 입력)
- 별도 테이블이면 N+1 조회 + 작성 시 트랜잭션 복잡도 ↑
- v1 클로즈드 베타에 단순화 우선. 추후 검색·통계 필요해지면 별도 테이블로 migration

**JSON shape (TypeScript 타입과 1:1)**:

```ts
// techTags
type TechTag = { label: string; tone?: "primary" | "mint" | "lav" | "peach" | string };

// metrics — 두 가지 패턴 동시 지원
type Metric =
  | { k: string; before: string; after: string }              // "TPS: 500 → 2,000"
  | { k: string; delta: string; dir: "up" | "down" };         // "월간 운영비: -₩2,000,000"
```

### 2.3 마이그레이션

- DB: PostgreSQL 16 (기존)
- 5.1 `User`/`ConsentLog`, 5.2 `Profile` 모두 Hibernate `ddl-auto` 로 자동 생성됨 (`application.yml` 확인). Career/Project 도 동일 패턴 — 별도 Flyway 스크립트 없음. prod 전환 시 별도 마이그레이션 도구 도입 결정 (별도 작업, 본 PR 범위 외).

### 3.2 POST /api/v1/me/careers — 회사 생성

요청:
```ts
type CareerCreateRequest = {
  companyName: string;
  position: string;
  startDate: string;
  endDate: string | null;
};
```
응답: 201 + 생성된 Career (without projects, displayOrder = max + 1).

### 3.3 PUT /api/v1/me/careers/{cid} — 회사 수정

요청: CareerCreateRequest 동일.
응답: 200 + 수정된 Career.

### 3.4 DELETE /api/v1/me/careers/{cid} — 회사 삭제

응답: 204. 연관된 Project 모두 cascade delete.

### 3.5 POST /api/v1/me/careers/{cid}/projects — 프로젝트 생성

요청:
```ts
type ProjectCreateRequest = {
  title: string;
  startDate: string;
  endDate: string | null;
  contributionPercent: number | null;
};
```
응답: 201 + 생성된 Project (PRAR 모두 null, techTags/metrics 빈 배열, displayOrder = max + 1).

### 3.6 PATCH /api/v1/me/careers/{cid}/projects/{pid} — 프로젝트 수정 (자동저장 진입점)

요청 (부분 업데이트 — 보낸 필드만 수정):
```ts
type ProjectPatchRequest = {
  title?: string;
  startDate?: string;
  endDate?: string | null;
  contributionPercent?: number | null;
  prar?: {
    problem?: string;
    rootCause?: string;
    approach?: string;
    result?: string;
  };
  techTags?: TechTag[];
  metrics?: Metric[];
};
```

응답: 200 + 수정된 Project 전체.

**PATCH** 사용 이유: onBlur 자동저장이 한 셀 (e.g. `prar.problem`) 만 보내는 케이스에 맞음. 다른 필드 미보내면 미변경.

**부분 업데이트 의미 명확화**:
- DTO 의 모든 필드 `Optional<T>` 또는 nullable wrapper 사용. 요청에 키 자체가 없으면 미변경, 키가 있고 값이 null 이면 null 로 설정.
- 단 `prar` 는 nested object — 통째로 보내야 함 (`prar: { problem: "x" }` 보내면 problem 만 업데이트, rootCause/approach/result 는 미변경). Jackson 의 `JsonInclude.NON_ABSENT` + nullable Optional 로 구분.
- `techTags` / `metrics` 배열은 **통째로 교체** (요청에 키 있으면 배열 전체 덮어쓰기, 없으면 미변경). PATCH 호출자는 항상 현재 배열 + 변경 적용한 결과를 보냄.

### 3.7 DELETE /api/v1/me/careers/{cid}/projects/{pid}

응답: 204.

### 3.8 에러 응답

기존 `GlobalExceptionHandler` 패턴 그대로:
- 404 `CAREER_NOT_FOUND` / `PROJECT_NOT_FOUND`
- 403 `FORBIDDEN` (다른 사용자 리소스 접근)
- 400 `VALIDATION_FAILED` (Bean Validation)

```tsx
function checkIssues(careers: Career[]) {
  const issues: string[] = [];
  careers.forEach((c) => {
    c.projects.forEach((p) => {
      if (p.metrics.length === 0) {
        issues.push(`"${p.title}" 프로젝트의 정량 성과가 비어있어요.`);

      const prarFilled = [p.prar.problem, p.prar.rootCause, p.prar.approach, p.prar.result].filter(Boolean).length;

      }
    });
  });
  return issues;
}
```

## 5. 자동저장 흐름 + 테스트

### 5.1 자동저장 (onBlur)

PrarCell:
1. textarea 입력 → 내부 state 업데이트 (controlled)
2. onBlur → parent 의 `onChange(localValue)` 호출
3. ProjectCard 의 onChange 가 `autosave({ prar: { [field]: localValue } })` 호출
4. PATCH 성공 → setProject(updated) (cascade re-render)
5. PATCH 실패 → top error banner 표시 + 사용자 재시도 가능

MetricChip / TechTag 추가·삭제:
1. + 클릭 → inline input 표시
2. Enter / blur → parent 의 `onChange(newMetrics)` 호출
3. autosave({ metrics: newMetrics })

**디바운싱 안 함** — onBlur 는 단일 이벤트라 호출 폭주 위험 없음.

**시각 피드백**:
- 저장 중 → 작은 spinner 아이콘 (PRAR cell 우상단)
- 저장 성공 → 잠시 체크 아이콘 fade (선택, v1 안 해도 OK)
- 저장 실패 → top error banner ("저장 실패. 잠시 후 재시도")

### 5.2 BE 테스트

`CareerControllerTest.java` (통합 테스트, MockMvc + DB):
1. GET /careers — 비인증 시 401
2. GET /careers — 인증 후 빈 배열
3. POST /careers — 생성 + DB 저장 확인 + displayOrder 자동 증가
4. POST /careers — Bean Validation 실패 (companyName 빈값) → 400
5. PUT /careers/{cid} — 자신의 회사 수정 성공
6. PUT /careers/{cid} — 타인의 회사 수정 시도 → 403

## 7. 비기능 / 운영

- **백엔드 마이그레이션**: PostgreSQL 16 + Hibernate auto-DDL. v1 클로즈드 베타라 별도 Flyway 미도입 — 기존 5.1/5.2 패턴 따름. 추후 prod 전환 시 마이그레이션 도구 통일 검토.
- **JSON 컬럼**: PostgreSQL jsonb. Hibernate 6 의 `@Column(columnDefinition = "jsonb")` + `@JdbcTypeCode(SqlTypes.JSON)` 사용.
- **접근성**: SideNav 의 disabled 항목 `aria-disabled="true"`. PRAR cell label 과 textarea `htmlFor`/`id` 연결.
- **번들 크기**: career.css ~870줄 (변환 후), /me/career 라우트에만 bundle.
- **i18n**: v1 한국어만. 정적 카피.
- **하이드레이션**: career-content.tsx 가 client 진입점. PrarCell 의 controlled state 는 client only.
- **데이터 사이즈**: 회사 1명당 ~10개, 프로젝트 회사당 ~5개 가정. 응답 크기 < 100KB.
