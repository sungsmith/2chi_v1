## 1. 배경 / 목표

§9 기업분석은 **회사 단위로 한 번 생성하면 여러 공고·자소서가 공유하는 분석 entity**. v1 의 핵심 가치는 (a) DART mock + 사용자가 입력한 회사 홈페이지 URL 을 LLM 으로 정제해 (b) 자소서·면접에 활용 가능한 인재상 키워드와 추천 포인트를 자동 생성하는 것. 와이어프레임 §9 의 5 카드 중 v1 은 3개만 (회사 개요·인재상·AI 활용 포인트), 채용 절차·뉴스는 v2 로 이월.

§9 의 결과는 §6 자소서 작성의 "기업분석 연결" select 에 노출되어 프롬프트 보강에 활용 (현재 5.7 머지 시점에는 disabled placeholder).

## 2. 결정 내역 (브레인스토밍 합의)

1. **데이터 소스 (v1)**: DART → mock fixture (실 Open DART API 호출 v2 이월), 회사 홈페이지 → 사용자 입력 URL 1~5개 (BE 가 jsoup 으로 스크랩), 뉴스 검색 → v2 이월.
2. **진입 경로**: (A) 별도 "기업" 메뉴 `/company/analysis` (목록·새 분석·상세) + (B) 5.5 공고 카드의 "🏢 기업분석" 버튼 (회사명 자동) + (C) §6 자소서 작성의 "기업분석 연결" select 활성화 (공고 회사명 일치 분석만 표시).
3. **v1 카드 구성** (3개): 🏢 회사 개요 (DART mock 의 overview · 사업 영역 · 매출 · 임직원 · 소재), 👥 인재상 (URL 스크랩 + LLM 추출한 5~7개 키워드 tag), 💡 AI 활용 포인트 (LLM 이 DART + 스크랩 + 사용자 경력 합쳐서 3~4개 자소서·면접 추천 생성).
4. **채용 절차 카드 v1 제거** — 데이터 소스 불명확. v2 에서 사람인 API 또는 사용자 입력 추가.
5. **뉴스 카드 v1 제거** (이미 결정).
6. **LLM 호출 1회**: 분석 생성 시 단일 prompt 로 talent_profile + action_points JSON 응답. gpt-4o-mini, ~$0.02/분석.
7. **30일 캐시 정책**: `generated_at` 기준 D-30 카운트다운 UI 표시. 자동 만료·재분석 안 함. 사용자 명시적 "🔄 재분석" 클릭만 trigger.
8. **`(user_id, company)` UNIQUE**: 같은 사용자가 같은 회사명을 두 번 분석하면 기존 row in-place 갱신. 분석 히스토리 보존 안 함 (v2).
9. **§6 select 매칭**: 자소서 작성 시 `posting.company == analysis.company` 정확 매칭. 회사명 정규화 v2.
10. **재사용**:
    - 5.7 의 `OpenAiCoverLetterClient` 패턴 그대로 모방 — 새 `OpenAiAnalysisClient`
    - 5.5 의 `JobkoreaParser` 의 jsoup 패턴 모방 — `HomepageScraperService`
    - `summary_json` 매핑은 5.7 의 `validation_json` 패턴 (`@JdbcTypeCode(SqlTypes.JSON)` + String)

## 5. API 설계

| Method | Path | 요청 | 응답 | 비고 |
|---|---|---|---|---|
| GET | `/api/v1/company-analyses` | — | `AnalysisSummaryResponse[]` | 목록 (generated_at desc) |
| GET | `/api/v1/company-analyses/{id}` | — | `AnalysisResponse` | 상세 |
| POST | `/api/v1/company-analyses` | `AnalysisCreateRequest` | 200 또는 201 `AnalysisResponse` | UNIQUE 충돌 시 in-place 갱신 (200), 신규 (201) |
| DELETE | `/api/v1/company-analyses/{id}` | — | 204 | hard delete |
| GET | `/api/v1/company-analyses/by-company?company=X` | query string | `ByCompanyResponse` | §6 select 용. 없으면 id=null |

`AnalysisCreateRequest`:
```ts
{
  company: string;          // @NotBlank @Size(max=200)
  urls: string[];           // 0~5 개. 각 URL @Size(max=1000), @Pattern http(s)
}
```

`AnalysisResponse`:
```ts
{
  id: number;
  company: string;
  summary: {
    overview: { businessArea: string; mainProducts: string[]; revenue?: string; employees?: number; location?: string };
    talent_profile: string[];        // 5~7
    action_points: string[];         // 3~4
  };
  sourceUrls: string[];              // 사용자가 입력한 URL 그대로 + DART mock 의 source url
  generatedAt: string;               // ISO 8601
  generatedBy: string;               // "gpt-4o-mini"
  expiresInDays: number;             // 30 - daysSinceGeneratedAt (음수 가능)
}
```

`AnalysisSummaryResponse` (목록·그룹용):
```ts
{ id: number; company: string; generatedAt: string; expiresInDays: number }
```

`ByCompanyResponse`:
```ts
{ id: number | null; company: string }
```

### Validation

- `company`: NotBlank, max 200
- `urls`: 0~5 size, 각 항목 max 1000자, http/https prefix 검증
- 사용자가 URL 0개로 분석 만들면 → DART mock 만으로 prompt 조립 (talent_profile 빈 배열 가능성)

{url1 본문 (최대 1500자)}

{url2 본문 (최대 1500자)}

다음 JSON 으로 정확히 반환 (다른 텍스트·코드블록·설명 금지):
{
  "talent_profile": ["키워드1", "키워드2", ...],
  "action_points": ["문장1", "문장2", ...]
}

규칙:
1. talent_profile: 회사 홈페이지 본문에서 추출한 인재상·핵심 가치 키워드 5~7개 (간결한 명사형). URL 입력 없으면 빈 배열.
2. action_points: DART + 홈페이지 + 사용자 경력을 묶어서 자소서·면접에 활용할 구체 추천 3~4개. 각 1~2문장. 1인칭, 해요체. 회사 특이성·정량 지표 강조.
3. JSON 외 출력 금지. 코드블록(```) 금지.
```

gpt-4o-mini, max_tokens 1500, temperature 0.5.

## 10. 미해결 / 위험

1. **DART mock fixture 의 회사 list**: 어느 회사를 mock 에 넣을지. v1 사용자 테스트 편의로 2~3개 (테크컴퍼니·삼성전자·네이버 등). 매칭 안 되는 회사는 generic `_default` fallback — UX 적으로 "DART 데이터 없음" 안내.
2. **회사 홈페이지 스크랩의 robots.txt·법적 이슈**: v1 은 사용자가 입력한 URL 만 fetch (능동 권한). 단, 명시적 robots.txt 위반 회피·timeout 5초·user-agent 명시 필요.
3. **OpenAI 응답이 JSON 외 텍스트 반환할 가능성**: gpt-4o-mini 가 system prompt 무시하고 코드블록 또는 설명 추가하는 경우. 응답 parse 시 ```json ... ``` 트림 + JsonProcessingException catch → ANALYSIS_GENERATION_FAILED.
4. **§6 select 의 analysisId 가 variant 저장 시 어떻게 활용?**: 5.7 의 createWithAiDraft 에서 req.analysisId 를 받지만, AiService.generateDraft 에는 전달 안 됨 — analysis 정보가 프롬프트에 반영 안 됨. v1 spec 의 의도: analysisId 는 metadata 로만 저장 (variant.analysis_id), 프롬프트 보강은 v1.1 (5.7 후속 task). 또는 본 PR 에서 함께 — Phase D 로 결정.
5. **§9 결과의 expiresInDays 계산**: BE 가 계산해서 응답 vs FE 가 generatedAt 으로 계산. BE 에서 계산해서 명시적 필드로 노출하는 게 깔끔.
6. **회사명 정규화 v2 의 영향**: 사용자가 "테크컴퍼니" 분석 후 "(주)테크컴퍼니" 공고 등록 → §6 select 매칭 안 됨. v1 에서 명시적 안내 ("회사명을 정확히 일치시켜야 분석이 자동 연결돼요") 또는 침묵.
