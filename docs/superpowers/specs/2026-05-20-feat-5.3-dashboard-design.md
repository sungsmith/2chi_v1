## 2. 접근법 — Server Page + Client DashboardContent

`(app)/page.tsx` 는 얇은 Server Component (기존 5.0 패턴 유지). 내부에 `<DashboardContent />` 를 "use client" 로 작성하여 `useAuth()` 로 /me 데이터 접근.

근거:
- 5.0/5.1/5.2 의 기존 인증 데이터 흐름과 일치 (AuthContext via client)
- (app)/layout 의 Server 가드와 Client AuthBoundary 패턴을 그대로 활용
- v1 클로즈드 베타에 서버 사이드 fetch /me 추가는 과잉 (YAGNI)

**거부한 대안**:
- 완전 Server Component — 쿠키→백엔드 fetch 패스, AuthContext 와 데이터 중복 위험
- 완전 Client Component — page.tsx 클라이언트화는 (app)/layout 의 가드 패턴과 불일치

## 5. 인사 헤더 (Greeting) 매핑

| 영역 | 소스 | 값/규칙 |
|---|---|---|
| `.greet-meta` 날짜 | 클라이언트 현재 시각 | 시안 표기 `2026 · 05 · 12 수요일 · 오늘의 준비 현황` 형식. `Intl` 기본 출력 (예: `2026. 05. 12. 수요일`) 을 그대로 쓰지 말고, 각 필드를 `formatToParts()` 로 추출 후 ` · ` 로 join → `${y} · ${m} · ${d} ${wd}`. 끝에 ` · 오늘의 준비 현황` 정적 텍스트 결합 |
| `<h1>` 닉네임 | `useAuth().user.nickname` | `안녕하세요, {nickname}님 👋` |
| `.line2` 카피 | 정적 | "오늘도 이취가 다가오는 일정과 작성 흐름을 같이 정리해드릴게요. 내 이력과 지원 현황을 기준으로, 이번 주에 챙기면 좋을 준비를 모아뒀어요." (시안 그대로) |
| `.greet-tags` 3개 | Mock | 백엔드 / 중고신입 (2년차) / 이직 준비 중 — **`showTags=true` 일 때만 렌더** |

**온보딩 미완료 분기**: `showTags=false` → 부제 태그 3개 숨김. 닉네임만 실데이터인 상태에서 직무·경력·목적이 Mock 으로 노출되면 사용자 혼동 → HomeBanner 가 "프로필 완성하세요" 안내 역할.

날짜 포맷의 "·" 구분자는 `Intl` 결과를 후처리(`replace(/\. ?/g, " · ")` 등)하여 시안 표기와 맞춤. 시각 회귀 위험 영역이라 `greeting.test.tsx` 에서 고정 시각 케이스로 검증.

## 7. CSS 통합

`design_system/pages/dashboard.css` (911줄) 을 `frontend/src/app/(app)/dashboard.css` 로 복사하되, 아래 4개 변환 필수. `page.tsx` 에서 `import "./dashboard.css"`.

이유:
- Next.js app router 는 페이지 단위 CSS import 지원 (해당 라우트에만 bundle)
- 시안의 클래스명(`.kpi`, `.greet`, `.sched-row`, `.shortcut.primary` 등)이 코드와 1:1 대응 → 시안 ↔ 구현 디버깅 단순
- CSS Modules 도입은 프로젝트 첫 모듈이라 학습 비용·디렉토리 복잡도 추가. 현재 frontend 가 globals + inline style + token 으로 운영 중이라 정합성도 떨어짐

**복사 시 변환 작업** (탐색 결과 기반):

1. **`.dash-nav` / `.dash-nav-inner` / `.dash-nav-right` 영역 삭제** (시안 L46~170)
   - 5.0 의 TopNav 가 자체 inline style 로 이미 렌더 중. dashboard.css 의 nav 스타일 적용 시 충돌·중복
   - dashboard.css 에서 해당 블록을 통째로 삭제 (`.dash-main` 위까지)

2. **`body.dash-body` 셀렉터 → `.dash-page` 래퍼 셀렉터** (L7~41)
   - Next.js app router 에서 body 클래스 부여는 root layout 수정 필요 (전역 누수). dashboard 한 페이지에 body 클래스는 과함
   - `body.dash-body { ... }` → `.dash-page { ... position: relative; }` (position: relative 추가하여 ::before/::after 가 fixed 가 아닌 absolute 로 떠도 컨텍스트 잡힘)
   - `body.dash-body::before { position: fixed; ... }` → `.dash-page::before { position: absolute; ... }` (페이지 스크롤과 함께 흐르지 않게 의도라면 fixed 유지하되 `.dash-page` 와 함께 동작 검증). 시안에서는 fixed 였으므로 일단 fixed 유지하고 시각 확인 후 결정
   - 동일하게 `body.dash-body::after` → `.dash-page::after`
   - `page.tsx` 의 `<section>` 에 `className="dash-page"` 부여

3. **`../tokens.css` / `../doc.css` import 라인** (시안 dashboard.css 에는 import 라인 없음 — 시안 dashboard.html 이 별도 link 태그로 로드)
   - 확인 결과 dashboard.css 자체에는 `@import` 가 없음 → 별도 작업 불필요
   - `doc.css` 의존 클래스 grep 결과: dashboard.css 는 `.doc` 을 `.stage.doc` (서류 단계 배지) 용도로만 사용 → doc.css 와 무관. doc.css 미통합

4. **`@font-face` / 폰트 import**: dashboard.css 에 없음 (시안 dashboard.html 의 link 태그가 폰트 별도 로드). globals.css 의 Pretendard / Caveat import 가 dashboard 에서도 유효 → 추가 작업 불필요

## 10. 비기능 / 운영

- **번들 크기**: dashboard.css 911줄(약 15~20KB). 해당 라우트에만 bundle (Next.js page-level CSS import). 다른 페이지 영향 없음
- **접근성**: disabled shortcut 에 `aria-disabled="true"` + 시각적 데세추레이트. 색상 대비는 시안 토큰 유지
